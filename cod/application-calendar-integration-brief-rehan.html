2. Why we're replacing GHL's calendar

The current flow is: form submit → leave our domain → land on app.clientsondemand.app/widget/bookings/breakthrough-call → pick a slot → confirm.

Two problems with the handoff:

  • Domain switch breaks the visual flow. Users perceive it as "the page reloaded" and a chunk bounce here.
  • The GHL widget styling does not match the form. The applicant just spent 90 seconds in our dark-gold experience and lands on a different-looking page. Trust erosion.

Compounding effect: The form rebuild attacks the form half of the 75% dropoff. The calendar integration attacks the handoff half. Together they should move the needle materially.

4. Cloud function spec (Phase 2 details)

A. Endpoint shapes

GET /api/cod/free-slots

Request:
  GET /api/cod/free-slots?date=2026-05-15&tz=America/Mexico_City
  Headers: none (CORS-validated by Origin)

Response (200):
  {
    "date": "2026-05-15",
    "timezone": "America/Mexico_City",
    "slots": [
      { "start_iso": "2026-05-15T09:00:00-06:00", "label": "9:00 AM" },
      { "start_iso": "2026-05-15T10:30:00-06:00", "label": "10:30 AM" }
    ]
  }

Response (200, no slots):
  { "date": "2026-05-15", "timezone": "...", "slots": [] }

Response (400, bad date):
  { "error": "invalid_date", "message": "..." }

Response (502, GHL down):
  { "error": "upstream_error", "message": "..." }

POST /api/cod/book

Request:
  POST /api/cod/book
  Content-Type: application/json
  Body:
  {
    "slot_iso": "2026-05-15T09:00:00-06:00",
    "first_name": "...",
    "last_name": "...",
    "email": "...",
    "phone": "...",
    "professional_category": "...",
    "professional_role": "...",
    "what_you_help_with": "...",
    "decision_style": "...",
    "sms_consent": true
  }

Response (200):
  {
    "booked": true,
    "appointment_id": "ghl_appt_...",
    "contact_id": "ghl_contact_...",
    "slot_iso": "2026-05-15T09:00:00-06:00",
    "calendar_invite_url": "..."   // optional, if GHL returns one
  }

Response (409, slot taken):
  { "error": "slot_taken", "message": "This time was just booked. Please pick another." }

Response (422, validation):
  { "error": "validation", "fields": { "email": "invalid" } }

5. UI integration spec (Phase 3 details)

Sequence at form submit

  1. User finishes Step 4 (contact info)
  2. Submit button shows the loading spinner (already in current code)
  3. POST to /api/cod/book happens at slot-confirm time — not at Step 4 submit. So Step 4 just transitions to Step 5 (calendar) without any GHL call yet.
  4. Step 5 calls GET /api/cod/free-slots for today + the next 6 days, populates the date strip
  5. User picks a date → fetch slots for that date → render time grid
  6. User picks a slot → enable Confirm button → on click → POST /api/cod/book with all form data + slot
  7. On 200 → show in-page success state (already designed in mock)
  8. On error → show error state, allow retry

This is different from today's hidden-GHL-form pattern. With the in-page calendar, the cloud function is the one calling GHL — the form no longer submits to GHL directly. Remove the hidden GHL form once Phase 3 is wired.

Important: Step 4 of the current form still has the hidden GHL form wired in for fallback. Keep that wiring intact until Phase 3 is done and verified. Only remove the hidden GHL form once the cloud function path is proven in production.

7. What to NOT do

  • Don't redesign the calendar UI. The Apr 23 mock is the design contract. Lift it as-is.
  • Don't put GHL API keys in client JS. Anything that hits GHL goes through the cloud function.
  • Don't break the existing form rebuild. The 4-step form is live and working. Add Step 5, don't refactor the first 4.
  • Don't ship without Matt's sign-off. Phase 2 has zero starting context until DISC-01 + DISC-02 are answered.
  • Don't cutover on a Friday. Soft launch midweek, cutover Tuesday or Wednesday.
