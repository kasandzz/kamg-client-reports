1. Why we are doing this

The current GHL-native application form has a 75% dropoff on the booking page. On mobile, picking a Step 1 option makes the form jump out of the viewport, looks broken, users bounce. GHL Forms cannot fix this -- their renderer controls height and scroll.

The redesign is a self-contained HTML/CSS/JS page that does not depend on GHL's form widget for layout. Your job is to make this the live page AND wire its output into GHL so all back-end automations (calendar, pipeline, emails, attribution) still work.

4. Funnel page setup in GHL

1

Clone the existing Application step

Sites → Funnels → Application Funnel → right-click the /application step → Clone → rename clone to application-v2. Do NOT touch the live /application URL yet.

2

Wipe the page body

Open application-v2 in Funnel Builder. Delete every section currently on the page (the GHL-native form, the headline section, footer -- everything). You want a blank canvas.

3

Add one full-width row + one Custom Code block

Drop in a single full-width row. Inside it: add a Custom Code element (under "Elements" panel). That is where the entire HTML/CSS/JS will live. Do not add anything else to the page.

5. Drop in the Custom Code block

Open the source file: application-form-redesign.html. View source (Ctrl+U). Copy everything from <!DOCTYPE html> to </html>. Paste into the Custom Code block.

GHL Custom Code constraint: GHL strips the <html>, <head>, and <body> tags. Paste it anyway -- it works -- but if anything breaks, strip everything outside <body>...</body> and re-paste just the inner content. Move the <link> font preloads + <style> into the Funnel page's "Tracking Code → Head" section.

Fonts: the design uses Cormorant Garamond (headings) + Outfit (body) loaded from Google Fonts. The <link rel="preconnect"> and <link href="https://fonts.googleapis.com/css2?family=Cormorant+Garamond..."> lines at the top of the file MUST stay. Without them the page will fall back to system fonts and look generic.

6. The hidden GHL form (this is the bridge)

The redesigned page is NOT a GHL form -- it is custom HTML. To get data into GHL contacts, you need a hidden GHL-native form on the same page that receives the data on submit.

1

Add a GHL form element below the Custom Code block

In Funnel Builder, drop a Form element below the Custom Code row. Build a form with these fields (use the Add Field panel, link each to the custom field key from Section 2):

  • First Name → standard contact field
  • Last Name → standard contact field
  • Email → standard contact field
  • Phone → standard contact field
  • Professional Category → professional_category
  • Professional Role → professional_role
  • What You Help With → what_you_help_with
  • Decision Style → decision_style
  • SMS Consent → sms_consent
2

Set the form ID and hide it

In the form's Advanced → Custom CSS, set:

#ghl-hidden-form { display: none !important; }

In the form settings → HTML ID field, set: ghl-hidden-form

The JS in the Custom Code block at line 1487 looks for document.getElementById('ghl-hidden-form'). If the ID is anything else, submission silently fails.

3

Match each input's name attribute to these exact strings

first_name      last_name      email      phone
professional_category   professional_role   what_you_help_with
decision_style          sms_consent

GHL usually does this automatically when you map the field to a custom field key, but verify in the Inspector (right-click the input on the published page → Inspect → check name="...").

7. Honeypot (hidden trap field for bots)

What it is: An invisible text field called website_url. Real humans never see it because it's positioned left:-9999px. Bots that crawl the DOM see "looks like a form field" and fill it in. If any value is in website_url on submit, we reject the submission silently and fire a PostHog event.

Where it lives in the HTML

Already in the source file at lines 1007-1011 inside the Step 4 panel. It's part of the visible form rendered to the page (not the hidden GHL form). The JS check at line 1443-1447 runs BEFORE the GHL form submits.

What you need to do in GHL

  • Nothing extra. The honeypot lives entirely in the Custom Code block. No GHL custom field needed.
  • Verify after publishing: View Source on the live page, search for website_url, confirm the wrapper div has style="position:absolute;left:-9999px;...".

How to test the honeypot works

  1. Open the published page in Chrome DevTools.
  2. Console: document.getElementById('website_url').value = 'http://spam.com'
  3. Fill the rest of the form normally. Click Submit.
  4. You should see NOTHING happen on the page. In the PostHog dashboard you should see a cod_application_honeypot_caught event with reason: honeypot_filled.

8. Timing trap (bots submit too fast)

What it is: A timer starts when the page loads (RENDER_TS = Date.now()). On submit, if less than 3000ms has elapsed, the submission is rejected as a bot. Real humans take at least a few seconds to read Step 1.

Configuration

const MIN_HUMAN_SUBMIT_MS = 3000; // line 1102

What you need to do in GHL

  • Nothing. It's already in the JS. Do not change the value.
  • If you ever see legitimate users getting blocked (very unlikely -- the form takes 30+ seconds to complete), tell Kas before increasing/decreasing.

How to test

  1. Open the page. Immediately open DevTools console.
  2. Run: document.getElementById('btn-submit').click() within 3 seconds. (You'd need to bypass the disabled state first -- this is a synthetic test.)
  3. Or just trust the code -- PostHog will fire cod_application_honeypot_caught with reason: too_fast if it triggers in production.

9. Draft autosave (localStorage, 24h expiry)

What it is: Every time the user types in a field or selects a card, their progress is saved to browser localStorage under the key cod_app_draft_v1. If they close the tab and come back within 24h, the form rehydrates and they pick up where they left off. After 24h, the draft auto-expires.

What gets saved

Category, role, decision style, help text, first name, last name, email, phone, SMS consent. NOT the honeypot value (obviously).

What you need to do in GHL

  • Nothing. All localStorage logic is in the JS (lines 1141-1196).
  • The draft is cleared automatically on successful submit, so a returning visitor after booking gets a fresh form.

How to test

  1. Fill out Steps 1, 2, 3 fully. Stop at Step 4.
  2. Close the tab.
  3. Reopen the page. You should see Steps 1-3 already filled when you navigate back through them, and the textarea char count should show your earlier text.
  4. To inspect: DevTools → Application → Local Storage → your domain → look for cod_app_draft_v1.

11. PostHog identify (tie session to contact)

What it is: On final submit, the page calls posthog.identify(email, {first_name, last_name, phone, role}). This converts the anonymous PostHog session into a tracked contact so every event the person ever fired (workshop watch, scroll depth, page views, etc.) is now stitched to their email and visible on their GHL contact.

What you need to do in GHL

  • Nothing extra in GHL. The identify call is already in the JS at lines 1473-1484. As long as the PostHog snippet from Section 10 is loaded BEFORE the form submits, identify will work.
  • Reference: Kas published rehan-posthog-identify-guide-2026-05-06.html -- you have it. Same pattern.

How to verify it works

  1. Submit the form as yourself (use a unique test email like rehan+app1@kasandz.com).
  2. In PostHog, search Persons for that email.
  3. You should see a Person record with first_name, last_name, phone, professional_role properties populated.
  4. Open the Person profile → you should see all the cod_application_step_* events for THIS submission listed under their timeline.

12. Hyros script

What it is: Hyros is COD's ad-attribution platform. It needs its own tracking script on every conversion page to attribute the booking back to the originating ad.

What you need to do

  1. Open GHL → Funnel → Settings → Tracking Code → Head Tracking Code (same place as PostHog).
  2. Paste the Hyros script. Get the exact snippet from the existing live application page (View Source) OR from any of the $27 funnel pages.
  3. The Hyros snippet looks roughly like:
<script async src="https://t.clientsondemand.com/v1/lst/universal-script?ph=..."></script>

Do NOT swap or remove the placeholder hash -- it identifies the COD account. Copy the existing one verbatim.

How to verify

  1. Open the published page. DevTools → Network tab → filter by "hyros" or "t.clientsondemand".
  2. Reload. You should see one request to the Hyros endpoint return 200.
  3. In Hyros dashboard, check that today's traffic shows the new /application-v2 URL.

13. Calendar passthrough URL (THIS IS CRITICAL)

The most-broken part of every form rebuild. On submit, the user needs to land on the Breakthrough Call calendar with their name/email/phone PRE-FILLED. If you do this wrong, every booking shows up blank in GHL and the sales team gets useless leads.

1

Open the hidden GHL form → On Submit settings

Click the hidden form → Settings → On Submit Actions.

2

Set "Redirect to URL"

Paste this EXACT URL into the redirect field:

https://app.clientsondemand.app/widget/bookings/breakthrough-call
  ?first_name={{contact.first_name}}
  &last_name={{contact.last_name}}
  &email={{contact.email}}
  &phone={{contact.phone}}
  &professional_role={{contact.professional_role}}
  &what_you_help_with={{contact.what_you_help_with}}
  &decision_style={{contact.decision_style}}

Strip the line breaks when you actually paste -- it should be one continuous URL. The line breaks above are for readability only.

3

Confirm the calendar group is unchanged

Matt's warning, repeated: if anyone clones or moves the calendar to a different calendar group, merge fields stop resolving and every booking lands with blank contact info. Verify the calendar at app.clientsondemand.app/widget/bookings/breakthrough-call is the same one used today before flipping traffic.

Do NOT redirect with JavaScript (window.location.href = ...). Merge tags {{contact.first_name}} will not resolve because GHL hasn't saved the contact yet at the moment JS runs. The redirect MUST be set via the GHL form's On Submit Action so GHL processes it server-side after contact creation.

14. Tags + pipeline stage on submit

Still in the hidden GHL form → On Submit Actions, ADD these (in addition to the redirect):

  1. Add Tag → application-submitted-v2 (lets us segment the new-page submissions in dashboards)
  2. Add Tag → step:applied (canonical funnel-stage tag we use everywhere)
  3. Update Pipeline Stage → move contact to "Applied -- Awaiting Booking" in the Application Pipeline

If the pipeline stage doesn't exist yet, ask Kas first -- don't create new pipeline stages without sign-off (they affect every dashboard).

15. A11y + mobile fixes + Trustpilot (already in code, just verify)

Already in the JS

  • All .card-option cards get role="button", tabindex="0", aria-pressed
  • Enter and Space keys trigger selection
  • Step indicator has role="progressbar" with live aria-valuenow
  • Phone field has inputmode="tel" (numeric keypad on mobile)
  • All inputs have proper autocomplete attributes (browser autofill works)

Mobile reflow fix (THE bug from current page)

  • Function recenterForm() runs after every step change
  • It calls scrollIntoView({behavior:'smooth', block:'center'}) on the active step
  • This fixes the "form disappears off screen after tapping option" bug on mobile Safari
  • Verify: on iPhone, tap any Step 1 category card. The next panel should auto-scroll into view.

Trustpilot widget (in footer)

Already in the HTML. Loads via tp.widget.bootstrap.min.js. Has a static fallback (4.4 / 5 stars, "500+ verified reviews") that displays if the widget fails to load within 3 seconds.

Verify after publishing: the real Trustpilot iframe should appear within ~2 seconds. If you only see the static fallback (no iframe), the widget script is blocked -- check GHL's tracking-code allowlist.
