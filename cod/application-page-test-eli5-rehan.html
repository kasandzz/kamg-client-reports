2. Why the current page is bleeding money

The bug: GHL's native form component re-heights itself when you pick certain options on Step 1. On mobile, the newly-rendered short form ends up above the scroll viewport — so the user sees a blank screen after tapping an answer and assumes the page is broken. They bounce.

The math: 75% dropoff on the booking page = 3 out of every 4 qualified leads who made it through the $27 workshop / VSL never book a breakthrough call. Fixing this is a bigger lever than anything upstream.

Why GHL forms suck for this specifically:

  • No control over post-selection scroll position
  • Dynamic height on multi-step = layout shift on every tap
  • No native "card-style" radio buttons (current form is dropdowns + labels → high cognitive load)
  • Mobile validation UX is weak

3. What we're replacing it with

Kas's redesign is a self-contained HTML/CSS/JS page at application-form-redesign.html. It's a 4-step wizard with card-based choices, fixed viewport, smooth slide transitions, and no dependency on GHL's form renderer. Rehan's job is to make this the live application page and wire its output into GHL so the back-end (calendar, pipeline, automations) still works exactly like today.

4. GHL-specific constraints Matt flagged

These are the things Matt has called out in the past about how GHL actually behaves under the hood. Respect them or this test will fail quietly.

A. Wiring form results into GHL fields

  • GHL will NOT auto-create a contact from a raw HTML form — it only recognizes submissions from its native <form> widget OR from an inbound webhook hitting https://services.leadconnectorhq.com/hooks/...
  • Easiest reliable path: build the custom HTML page in GHL's Funnel Builder → Custom Code block, but use GHL's native <form> element as the hidden submission layer. JS at submit time copies formData into the hidden GHL form inputs, then calls hiddenForm.submit().
  • Every custom field (all 5 from the table in section 3) must exist in Settings → Custom Fields before you build the page, otherwise GHL silently drops them.
  • Use field keys exactly as listed — Matt has warned before that GHL is case-sensitive and trailing spaces in labels cause "no such field" failures.

B. URL passthrough to the calendar

On submit, the user needs to land on the Breakthrough Call calendar with their name / email / phone already filled in. Matt's pattern for this:

https://app.clientsondemand.app/widget/bookings/breakthrough-call
  ?first_name={{contact.first_name}}
  &last_name={{contact.last_name}}
  &email={{contact.email}}
  &phone={{contact.phone}}
  &professional_role={{contact.professional_role}}
  &what_you_help_with={{contact.what_you_help_with}}
  &decision_style={{contact.decision_style}}
  • Set this as the Redirect URL in the hidden GHL form's "On Submit" settings. GHL replaces the merge tags server-side after the contact is created/updated.
  • The calendar widget will prefill name/email/phone automatically from the query string — that's built-in GHL behavior as long as the parameter names match the contact field keys.
  • Do not use window.location.href from JavaScript — merge tags won't resolve because GHL hasn't yet saved the contact at the moment the redirect fires. Always go through GHL's native redirect.
Gotcha Matt flagged: the calendar must be in the same calendar group it's in today. If you clone or move it, merge fields stop resolving and every booking lands with blank contact info.

C. Name passthrough has broken before

On the $27 workshop funnel earlier this month, VIP emails went out with empty {{contact.first_name}} tags because the page wasn't saving the name. Root cause was a custom-built input that wasn't mapped to the GHL contact field. Avoid repeating that:

  • The Full Name field in the redesign must be split into first_name + last_name before submit (use the first space as the split point).
  • Submit a test as yourself and verify both fields appear on the contact detail view in GHL before going live.

5. Step-by-step build order

1

Grab the source HTML

Download application-form-redesign.html from kamg-client-reports (the live link in the Slack thread). This is the design contract — don't restyle it. Preserve the Cormorant Garamond + Outfit fonts, the gold accent (#c8a96e), the card radii, and the step dots.

2

Create the 5 custom fields in GHL

Settings → Custom Fields → New. Names and types:

professional_category
Single line — text
professional_role
Single line — text
what_you_help_with
Multi line — text
decision_style
Dropdown (3 values: solo / spouse / business-partner)
sms_consent
Checkbox (boolean)
3

Clone the existing application funnel step

Sites → Funnels → Application Funnel → clone the /application step → rename to application-v2 (draft slug) so we can A/B against the live page without breaking traffic.

4

Replace the page body with a Custom Code block

Delete all existing sections. Drop in one full-width Custom Code block. Paste Kas's HTML/CSS/JS inside. At the bottom, inject a hidden GHL form element (built using GHL's form builder, styled display:none) that contains all 5 custom fields + name/email/phone. Wire the real "Submit" button's submitForm() function to:

function submitForm(){
  // existing formData capture…
  const hf = document.getElementById('ghl-hidden-form');
  hf.querySelector('[name="first_name"]').value = formData.name.split(' ')[0];
  hf.querySelector('[name="last_name"]').value  = formData.name.split(' ').slice(1).join(' ');
  hf.querySelector('[name="email"]').value      = formData.email;
  hf.querySelector('[name="phone"]').value      = formData.phone;
  hf.querySelector('[name="professional_category"]').value = formData.category;
  hf.querySelector('[name="professional_role"]').value     = formData.role;
  hf.querySelector('[name="what_you_help_with"]').value    = formData.description;
  hf.querySelector('[name="decision_style"]').value        = formData.decision;
  hf.querySelector('[name="sms_consent"]').checked         = formData.smsConsent;
  hf.submit();
}
5

Set the redirect URL on the hidden form

In the hidden GHL form's settings → On Submit → Redirect to URL → paste the URL-passthrough string from section 4B above. Test by filling out the page as yourself and confirming the calendar loads with your name and email prefilled.

6

Add tags + pipeline triggers

Still on the hidden form → Actions:

  • Add tag application-submitted-v2
  • Add tag step:applied
  • Move contact into pipeline Applied — Awaiting Booking

These tags let us segment the A/B test cleanly in the dashboard later.

7

Ship it to a test URL, not production

Publish to clientsondemand.live/application-v2 first. Do NOT overwrite /application until QA passes. Kas will decide when to flip the A/B split.

6. QA checklist (must all pass before Kas flips traffic)

  • ✓ On mobile (iPhone Safari + Chrome Android), tapping ANY option on Step 1 — including the bottom options — does NOT cause the form to disappear or jump scroll position.
  • ✓ Full 4-step flow completes without horizontal scroll on a 375px viewport (iPhone SE).
  • ✓ Phone field accepts + formats US numbers (123) 456-7890.
  • ✓ Submit button is disabled until name, valid email, and 10-digit phone are all filled.
  • ✓ After submit, a new contact appears in GHL with ALL 5 custom fields + first_name + last_name + email + phone populated.
  • ✓ After submit, user lands on the Breakthrough Call calendar page with name/email/phone already filled in (verify by reading the prefilled widget, not just the URL).
  • ✓ Tag application-submitted-v2 visible on the contact.
  • ✓ Contact has moved into Applied — Awaiting Booking pipeline stage.
  • ✓ Booking confirmation email fires correctly with {{contact.first_name}} resolved (NOT blank — this was the bug from the $27 funnel earlier).
  • ✓ Fonts (Cormorant Garamond + Outfit) load on first paint, no FOUT.
  • ✓ Page matches the visual style of the front-end funnel page (dark gold aesthetic).
