What's running, in numbers

One Cloud Function, on a 6-hour schedule, pulls 17 surfaces of GHL data into the COD warehouse, ready to be joined against Hyros, Stripe, Meta, and the rest of the stack.

04 / Bridges

How GHL connects to the rest of the warehouse

No physical bridge tables yet. Bridges today are email-based joins — every surface uses the contact's email as the join key. The master journey table will eventually collapse these into one event log per contact.

GHL ↔ Hyros (revenue attribution)

ghl_contacts + fact_payments → join on email → hyros revenue
Why: Hyros is the authoritative revenue source. GHL holds the contact context (profession, channel, attribution). Joining on email lets us validate Hyros payments against GHL orders and surface mismatches.

GHL ↔ Hot Hawk (behavioral signals)

ghl_contacts → join on email → hot_hawk events
Why: Hot Hawk captures behavioral signals (scroll depth, video watch time, page intent). Joining to GHL contacts lets us segment behavior by profession, budget capacity, and pipeline stage. Currently pending Hot Hawk API access.

GHL ↔ Enrollment Transcripts (semantic search)

ghl_contacts → join on email → enrollment_transcripts
Why: Enrollment-call transcripts (with Vertex embeddings, 768-dim) live in the cod-marketing-warehouse project for semantic search on prospect voice. Joining lets us pull "what did this person actually say on their enrollment call" alongside their pipeline data.

Master Journey Table PLANNED · P0

ghl_contacts + fact_bookings + fact_payments + hyros + hot_hawk → contact_journey (one row per event, per contact)
Why: Today, each surface is queried separately. The master journey table collapses every event for a contact into a unified event log so you can answer "show me every touch this person had, in order" with one query. This is the next high-leverage build on the warehouse roadmap.
05 / Validity Steps

What protects the data quality

These are the guards baked into ghl-sync that keep junk out and make sure broken syncs are visible, not silent.

✓ RATE-LIMIT GUARD

650ms baseline + 5× retry on 429

Every GHL call waits RATE_DELAY_MS = 650 after the previous one. If GHL returns a 429, we back off exponentially (2s → 4s → 8s → 16s → 30s cap) for up to 5 retries before failing the request. Campaigns sync runs first to preserve budget.

✓ TEST-RECORD FILTER

Regex strips dev/QA noise

Patterns include freemanananan, ai chat, test@, noreply, and example.com. Internal QA records stay out of the warehouse so dashboards never count them as real prospects.

✓ TIMESTAMP NORMALIZATION

Handles ISO8601, Excel serial, US dates

GHL exports return inconsistent date formats across endpoints. The sync normalizes to ISO 8601 / BigQuery TIMESTAMP and parses Excel serial numbers, US-format dates, and ISO strings into one consistent type.

✓ CURRENCY PARSING

Strips $ and commas to FLOAT64

Amounts come through as strings with currency symbols. Sync coerces to numeric so SUM / AVG queries work without per-query CAST gymnastics.

✓ DUPLICATE-PAYMENT DETECTION

2-minute window flag

Identifies near-duplicate transactions (same email + same amount within 2 minutes) and flags them so analytics can dedupe at query time without losing the raw record.

✓ ETL RUN LOG

Every sync, every table, audited

On every run, each table writes a row to etl_run_log with run_id, table_name, status, rows_loaded, duration_ms, started_at, completed_at, and error_message. One query tells you "is the pipe healthy" or "which table failed when".

✓ PHASE-ORDERED EXECUTION

Dimensions before facts

Reference data (users, products, custom fields, trigger links) syncs first. Then the cfMap built from custom fields enriches contacts and other facts on the second pass — so attribution columns are populated, not null.

✓ PER-REQUEST TIMEOUT

120s hard cap per call

Every HTTPS request is bounded by a 120s timeout. A hung GHL endpoint can't stall the whole sync — the request errors, retry logic engages, and the rest of the pipeline keeps moving.

06 / Problems Fixed & Open

The work behind the build

Reliability problems we've worked through, plus what's still open and who owns it.

Fixed

429 rate-limit storms on parallel table syncs
FIXED · ghl-sync v1.0
Problem: GHL's rate limit is shared across all calls for the location, not per-endpoint. Running contacts / bookings / forms in parallel exhausted the budget and high-volume endpoints returned 100% 429s.
Fix: Phase-ordered execution. Low-volume dimensions (campaigns, users, products) sync first to claim budget. 650ms baseline delay between requests. Exponential-backoff retry on 429. Campaigns specifically scheduled first because it's low-volume but rate-limit sensitive — running it after contacts cooked the budget.
Inconsistent timestamp formats across endpoints
FIXED · ghl-sync v1.0
Problem: GHL returned ISO 8601 in some endpoints, Excel serial numbers in others, US-format strings in a third set. Direct loads into BQ TIMESTAMP columns failed.
Fix: Normalization layer in ghl-sync that detects format and emits a single ISO string. Plus the v_fact_bookings_clean view that re-parses any messy call_date values that slipped through.
Test records polluting funnel metrics
FIXED · ghl-sync v1.0
Problem: Internal QA contacts (freemanananan, test@, noreply, example.com, AI-chat dummies) were counted as real prospects in dashboards.
Fix: Regex filter at sync time. Test rows never enter the warehouse — cleaner than filtering at query time on every dashboard.
Silent sync failures
FIXED · ghl-sync v1.0
Problem: Early version had no per-table audit. A failure on one of 17 tables wasn't visible until someone noticed stale data on a dashboard.
Fix: etl_run_log table. Every table-sync writes a row with status, row count, duration, and error message. One query = SELECT * FROM cod_warehouse.etl_run_log ORDER BY started_at DESC LIMIT 50 tells you which table is broken and why.

Open

"Appointment Status" custom field not visible in workflow builder
OPEN · raised 2026-03-20
Problem: Request to build a workflow that updates the "Appointment Status" custom field when a contact moves past the "Showed" pipeline stage. The workflow is built, but "Appointment Status" doesn't appear in GHL's workflow builder field list — the action shows a red error.
Action: determine whether the field is system-level vs custom, fix the field-mapping action, verify with a test contact move, sign-off before activation.
Workshop sheet — repeat purchases overwrite instead of appending
IN PROGRESS · raised 2026-03-31
Problem: The GHL workshop sheet automation updates an existing row when a contact buys a second time, instead of appending a new row. This corrupts repeat-attendance counts and obscures real funnel volume.
Status: Hyros is now the authoritative source for purchase data, so downstream revenue reporting is unblocked. Workshop sheet repair still owed for the secondary operational view of repeat attendance.
GHL API platform limits (no workflow / activity history)
PLATFORM LIMIT · ongoing
Problem: GHL API v2 doesn't expose workflow execution history, activity timeline, or analytics endpoints. Reconstructing "what happened to this contact" requires 7+ separate API calls (conversations, notes, tasks, appointments, opportunities, payments, forms, surveys) and stitching client-side.
Workaround: The 17 tables we sync cover everything the API does expose. The master journey table will be the stitched view. Notes, tasks, and recordings are excluded — fetching them takes 14h per run, so they live in a separate slower job.
max-instances=1 not configured in deploy
CONFIG
Problem: If two scheduler runs overlap, both hit GHL simultaneously and burn the shared rate-limit budget twice. The fix is --max-instances=1 on the Cloud Function, but the codebase has no deploy.sh committed — referenced in the header comment but missing from the directory.
Action: add a deploy.sh with gcloud functions deploy ghl-sync --max-instances=1 ... and commit alongside the Cloud Function source. Currently a manual GCP-console flag, which means it can drift if anyone redeploys without knowing.
07 / How To Use This Data

Sample queries that answer real questions

Three patterns Matt can copy into BigQuery to validate the pipe is working and pull the data you need.

Is the sync healthy?

-- Most recent run for every table, with row count and any errors
SELECT
  table_name,
  status,
  rows_loaded,
  duration_ms,
  started_at,
  error_message
FROM `green-segment-491604-j8.cod_warehouse.etl_run_log`
WHERE started_at > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 24 HOUR)
QUALIFY ROW_NUMBER() OVER (PARTITION BY table_name ORDER BY started_at DESC) = 1
ORDER BY started_at DESC;

Workshop attendees who haven't yet booked

-- Form submitters in last 14 days with no booking on file
SELECT
  c.email,
  c.profession_dropdown,
  c.budget_capacity,
  fs.page_url,
  fs.`timestamp` AS form_submitted_at
FROM `green-segment-491604-j8.cod_warehouse.ghl_form_submissions` fs
JOIN `green-segment-491604-j8.cod_warehouse.ghl_contacts` c
  ON c.email = fs.email
LEFT JOIN `green-segment-491604-j8.cod_warehouse.fact_bookings` b
  ON b.email = c.email
WHERE fs.`timestamp` > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 14 DAY)
  AND b.booking_id IS NULL
ORDER BY fs.`timestamp` DESC;

Pipeline value by closer, this month

-- Open opportunity dollar value per closer, current month
SELECT
  closer,
  COUNT(*) AS open_opps,
  ROUND(SUM(monetary_value), 0) AS pipeline_dollars
FROM `green-segment-491604-j8.cod_warehouse.fact_opportunities`
WHERE status = 'open'
  AND DATE(created_at) >= DATE_TRUNC(CURRENT_DATE(), MONTH)
GROUP BY closer
ORDER BY pipeline_dollars DESC;
