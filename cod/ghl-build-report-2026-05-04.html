03 / Data Sources

What each cloud function does

Ten ingestion functions feed the COD warehouse, each on its own cadence with platform-specific rate-limit, retry, and dedup behavior.

ghl-sync

17 tablesevery 6h

Pulls all CRM surfaces from GoHighLevel: contacts (with attribution columns extracted), opportunities, bookings, payments, orders, transactions, subscriptions, conversations, forms, surveys, workflows, campaigns, funnels, plus reference data. Rate-limit aware with 650ms baseline and 5× retry.

contactsopportunitiesbookingsorders+13 more

stripe-sync

3 tablesevery 6h

Charges, refunds, disputes. Recovers email via Stripe Customer object when the charge-level field is null. Detects $27 + $27 VIP combos (workshop + upgrade within 48h) so analytics can model the upsell as one event.

stripe_chargesstripe_refundsstripe_disputes

hyros-sync

2 tablesevery 6h

Server-side ad attribution truth. Pulls Hyros's conversion graph and revenue attribution tables. Hyros is the system-of-record for "which ad drove which sale" at ~98% accuracy via CAPI integration with Stripe.

hyros_conversion_datahyros_revenue_attribution

meta-ads-sync + meta-geo-sync

2 tablesevery 6h + daily

Daily ad-level performance from both COD ad accounts (#5 active, #2 historical). 14-day chunked requests with delete-and-insert dedup to avoid Meta API timeouts. Geo-sync covers campaign-level country / region spend and impressions.

meta_ad_performancemeta_geo_performance

sendgrid-sync

14 tablesevery 6h

Full SendGrid extraction: stats, campaigns, contacts, suppressions, templates, plus device, browser, geo, and mailbox-provider engagement breakdowns. 12-month rolling window; category stats go back 365d.

sendgrid_statssendgrid_campaignssendgrid_contacts+11 more

emailbison-sync

4 tablesevery 6h @ :15

Cold email outbound: campaigns, leads, replies, sender emails. Full-truncate refresh every run so the warehouse always reflects EmailBison's current state. Reply text capped at 5K chars to keep table widths sane.

bison_campaignsbison_leadsbison_repliesbison_senders

authnet-sync

2 tablesevery 6h, 180d window

Authorize.net (Deposyt HT account) transactions and settlements. 180-day rolling window. Important because Authorize.net bypasses Hyros entirely — without this sync, a chunk of revenue goes invisible to attribution.

authnet_transactionsauthnet_settlements

posthog-events-sync

3 tablesevery 6h @ :45

Web behavioral data: events, persons, sessions. Pageviews, custom events, autocapture, session metadata. Tuned page size (1000) and 7-day query window to dodge PostHog's 504 timeout under heavy load.

posthog_eventsposthog_personsposthog_sessions

05 / Cross-Source Bridges

How the sources connect

Four bridges are live in BigQuery today. Four more are spec'd or planned. Email is the universal join key; secondary keys are contact_id, stripe_customer_id, and phone.

● Live in BigQuery

identity_graph — universal person resolver

LIVE · 459K ROWS
Stripe+ GHL+ Hyros+ SendGrid+ PostHog → resolved on email → identity_graph (one row per person)
Answers: "Who is this person across every system?" Primary key is email, with phone, ghl_contact_id, and stripe_customer_id as secondary identifiers. Every other bridge joins through here.

bridge_sales_unified — unified revenue across 5 platforms

LIVE · VIEW
Stripe+ Hyros+ GHL payments+ Fanbasis+ Ontraport → join on email + date → unified revenue (Stripe wins ties)
Answers: "Did this person pay, and which platform recorded it?" ROW_NUMBER dedup with Stripe taking priority on cross-platform ties (e.g., a charge appearing in both Stripe and Hyros).

bridge_customer_journey — milestone timeline per contact

LIVE · VIEW
All payment sources → join on email → contact_journey (milestone view)
Answers: "What's this person's full revenue journey, in order?" Powers the journey explorer page in the dashboard. Some optional fields still null for contacts whose Authorize.net charges flow through GHL only.

bridge_data_reconciliation — Stripe vs Hyros monthly reconciliation

LIVE · VIEW
Stripe monthly vs Hyros monthly → join on month + product → reconciliation deltas
Answers: "Why do attribution and payment platforms disagree on revenue?" The Hyros gap by month: Dec 86.9%, Jan 83.1%, Feb 83.9%, Mar 3.8% — driven by Authorize.net / Affirm / manual charges that bypass Hyros's pixel.

● Spec'd or planned

bridge_ad_exposure_per_contact — which ads each contact saw

BLOCKED
Hyros+ Meta Ads → join on email + ad_id → contact-level ad exposure
Answers: "Which ads did this person see before converting?" Blocker: currently aggregate-only, not joined at the contact level. Tied to the blank "Ad" column in 4K+ registrants — fixing that unlocks creative-level ROAS attribution.

bridge_call_engagement — call attendance + outcome per booking

BLOCKED
Fathom recordings+ GHL bookings → join on booking_id + email → call engagement
Answers: "Did this person attend their call, and what was the outcome?" Blocker: Zoom CF deployed but returns 0 rows. Fathom API integration not yet pulling. VIP attendance tracking is live separately but not yet stitched into master_customers.

bridge_behavioral_scorecard — engagement score per contact

BLOCKED
PostHog events → join on email → behavioral score in master_customers
Answers: "How engaged was this person on our site?" Blocker: PostHog events live in customer_events but zero behavioral columns surface in master_customers. Depends on posthog.identify() getting wired up site-wide so person_id populates.

bridge_retention_metrics — subscription health + churn

PLANNED · PHASE 11
GHL subs+ Stripe subs → join on email + subscription_id → retention timeline
Answers: "Is this person still active? When did they churn?" Adds months_active, payment_failures, churn_date, tier_upgrades — none of which exist in the warehouse today.

07 / Source of Truth

Which system wins, by data domain

When two systems disagree, this is the resolution order. Most disagreements are Hyros-vs-Stripe (attribution vs payment) — both can be right at the same time. They measure different things.

Data domain Authority Secondary Notes
Ad-to-sale attribution Hyros (server-side CAPI, ~98%) Facebook Conversions API Hyros only counts pixel-attributed sales — anything that bypasses (Authorize.net, Affirm, manual) won't appear here.
Revenue transactions Stripe / Authorize.net / Fanbasis Hyros Whoever processed the payment is the truth on whether the dollar moved.
CRM / lead context GHL SendGrid (~265K contacts for retargeting Phase 10) Profession, budget, niche, attribution columns. Richest context on a contact.
Ad spend Meta Ads Manager Hyros Hyros pulls from Meta but Meta's number is the spend ground truth.
Landing-page behavior PostHog (via GTM) GHL form_submissions Page-level dwell, scroll, click — none of which GHL captures.
Enrollment / back-end product Ontraport GHL Back-end enrollments and tier upgrades live in Ontraport, not the front-end CRM.
Payment financing Fanbasis GHL payment_method field Five BNPL providers tracked: Affirm, ClarityPay, Payva, CreditKey, PayPal Later.
08 / Validity Steps

What protects the data

Guards baked into the cloud functions and views that keep junk out, surface failures, and reconcile cross-source disagreements.

✓ ETL RUN LOG

Every sync is audited

Every cloud function writes to etl_run_log with run_id, table_name, status, rows_loaded, duration_ms, started_at, completed_at, error_message. One query = warehouse health check.

✓ RATE-LIMIT GUARDS

650ms baseline + 5× retry

GHL sync: 650ms inter-request delay, 429 retry up to 5× with exponential backoff (2s → 30s cap). Meta sync: 14-day windowed requests to avoid timeouts. PostHog: 1000-row page size + 7-day window.

✓ TEST-RECORD FILTER

QA pollution stopped at the gate

Regex strips dev / QA contacts at sync time — patterns include freemanananan, test@, noreply, example.com, AI-chat dummies. Test rows never enter the warehouse.

✓ TIMESTAMP NORMALIZATION

One format out, many formats in

GHL endpoints, Meta API, and Sheets all return different date formats. Ingestion + the v_fact_bookings_clean view normalize ISO 8601, Excel serial, US-format, and human-readable strings into a single TIMESTAMP type.

✓ DUPLICATE-PAYMENT DETECTION

2-minute window + cross-source dedup

Stripe: detects $27 + $27 VIP combos within 48h and labels them. GHL: flags near-duplicate transactions (same email + amount within 2 min). bridge_sales_unified: ROW_NUMBER dedup, Stripe wins on ties.

✓ EMAIL RECOVERY

Stripe customer lookup on null email

Stripe charge-level email is null on ~36% of charges. Sync reaches to the Stripe Customer object to recover the email so the row can join through the identity_graph. Still leaves a gap — flagged below in Issues.

✓ DEDUP STRATEGY PER SOURCE

Right tool for right load shape

GHL / SendGrid / Authorize.net: WRITE_TRUNCATE (full refresh). Meta Ads: delete-and-insert per date partition. EmailBison: full truncate. Stripe: append + dedup at query time. Sheets: idempotent DELETE-today + INSERT.

✓ RECONCILIATION VIEW

Stripe vs Hyros, monthly

bridge_data_reconciliation exposes the gap between Stripe revenue and Hyros-attributed revenue per month. Drives the "why does Hyros say less than Stripe" conversation with explicit numbers (Feb: 83.9% gap, Mar: 3.8% gap).

10 / Problems Fixed & Open

The reliability ledger

What we've fought through to get here, and what's still on the critical path.

Fixed

429 rate-limit storms across multiple sources
FIXED · all syncs
Problem: Parallel API calls (especially GHL with shared rate budget) hit 100% 429s and stalled syncs.
Fix: Per-source rate guards — GHL 650ms + 5× retry + campaigns first; Meta 14-day chunked windows; PostHog 1000-row pages + 7d window. Phase-ordered execution ensures dimension data syncs before fact data.
Inconsistent timestamp formats across endpoints
FIXED · GHL + Sheets
Problem: GHL returned ISO 8601, Excel serial, and US-format strings in different endpoints. Sheets booking dates were free-text.
Fix: Normalization at sync time + the v_fact_bookings_clean view that re-parses any messy call_date values that slip through.
Test records polluting funnel metrics
FIXED · GHL sync
Problem: Internal QA contacts counted as real prospects in dashboards.
Fix: Regex filter at sync time. Test rows never enter the warehouse.
Stripe email-null on ~36% of charges (partial)
PARTIAL FIX · stripe-sync
Problem: 18,827 of 52,255 Stripe charges have null email — block on joining to identity_graph.
Fix: Email recovery via Stripe Customer object lookup. Recovers the majority of cases. Residual nulls remain — flagged below as critical blocker.
$27 + $27 VIP one-click upsell double-counted
FIXED · stripe-sync
Problem: Two $27 charges within 48h were counted as either $54 or 2×$27 depending on which downstream report you asked. Distorted blended CAC.
Fix: stripe-sync now detects the combo, labels both rows, and exposes the flag for downstream reports to dedupe consistently.

Critical blockers (open)

GHL Auth — 3 tables fail 401 every 6h
P0 · CRITICAL
Problem: Three GHL endpoints return 401 on every scheduler run. Surfaces as red rows in etl_run_log on every cycle. Suggests scoped-token mismatch between PIT permissions and the endpoints called.
Action: audit the PIT scope, regenerate with the missing permission, redeploy.
PostHog session_id 100% null
P0 · BLOCKS BEHAVIORAL
Problem: No session-level analysis is possible because every event has a null session_id. Person identification depends on posthog.identify() firing site-wide, which currently doesn't happen.
Action: wire posthog.identify(email) on the form-submit and post-checkout handlers. Until then, MA / VSL gauges and bridge_behavioral_scorecard stay empty.
Stripe vs Hyros — Mar 2026 gap collapsed to 3.8%
RESOLVED · ongoing monitoring
Problem: Dec / Jan / Feb 2025 the gap was 83-87% — Hyros pixel only saw a fraction of total revenue because Authorize.net, Affirm, and manual flows bypassed the pixel entirely.
Resolution: Authorize.net direct sync (authnet-sync) plus the Hyros CAPI integration cut the gap to 3.8% in Mar. bridge_data_reconciliation monitors the delta monthly so any regression is visible same-day.
Workshop sheet repeat-purchase overwrite
IN PROGRESS
Problem: Workshop sheet automation overwrites instead of appending on repeat purchases. Distorts repeat-attendance counts.
Status: Hyros now authoritative on revenue truth so reporting is unblocked. Sheet repair pending for the operational view of cohort attendance.
"Appointment Status" custom field invisible in workflow builder
OPEN · since 2026-03-20
Problem: Pipeline-stage → custom-field automation built but blocked because "Appointment Status" doesn't appear in GHL's workflow builder field list. Workflow action shows red error.
Action: determine if the field is system-level vs custom, fix the mapping, verify with a test contact move.
Zoom CF deployed but returns 0 rows
P1 · BLOCKS CALL ENGAGEMENT
Problem: Cloud Function exists for Fathom / Zoom call data but it isn't producing rows. bridge_call_engagement is blocked behind this.
Action: debug ingestion path, verify the API token / scope, confirm the target meeting list is non-empty.
Closer concentration risk — $10K vs $20K LTV uncertainty
P0 · WAREHOUSE GAP
Problem: Most enrollments funnel through one closer, and LTV varies $10K vs $20K depending on the conversion path. The warehouse can't yet attribute LTV to the closer path, so the gap stays a debate instead of a query.
Action: retention bridge (Phase 11) plus closer-attributed LTV columns in master_customers. Then the answer becomes a query, not a debate.
max-instances=1 not configured in deploy
CONFIG · P2
Problem: Two scheduler runs overlapping = double rate-limit consumption. Codebase references a deploy.sh that doesn't exist.
Action: commit deploy.sh with --max-instances=1; document the deploy command instead of relying on console memory.
11 / How To Use This Data

Queries that answer real questions

Three patterns to copy into BigQuery. Replace the project / dataset if running outside GCP.

Is the warehouse healthy right now?

-- Latest sync status per table, last 24h
SELECT table_name, status, rows_loaded, duration_ms, started_at, error_message
FROM `green-segment-491604-j8.cod_warehouse.etl_run_log`
WHERE started_at > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 24 HOUR)
QUALIFY ROW_NUMBER() OVER (PARTITION BY table_name ORDER BY started_at DESC) = 1
ORDER BY status DESC, started_at DESC;

Per-contact unified revenue across all platforms

-- One row per email with totals from every payment source
SELECT mc.email, mc.total_revenue, mc.payment_methods,
       mc.first_touch, mc.last_touch, mc.days_to_conversion
FROM `green-segment-491604-j8.cod_warehouse.master_customers` mc
WHERE mc.total_revenue > 0
ORDER BY mc.total_revenue DESC
LIMIT 100;

Stripe vs Hyros gap, this month

-- Reconciliation view — gap = Stripe revenue not attributed by Hyros
SELECT month, stripe_revenue, hyros_attributed_revenue,
       ROUND((stripe_revenue - hyros_attributed_revenue) / stripe_revenue * 100, 1) AS gap_pct
FROM `green-segment-491604-j8.cod_warehouse.bridge_data_reconciliation`
WHERE month >= DATE_TRUNC(CURRENT_DATE(), MONTH)
ORDER BY month DESC;
