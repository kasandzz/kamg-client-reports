01 ·HTML Block Placement & Editor Quirks

Where the editor itself fights you. Order, dropdowns, and built-in elements that look harmless but break in production.

01.01"Add to Calendar" buttons don't work in GHLDon't include

The native GHL "Add to Calendar" widget is unreliable across timezones and mobile clients. We've burned hours debugging it. It silently fails for ~30% of mobile users.

Fix: Strip the button from the design entirely. Calendar invites go out through the workflow email automation post-booking, not from the page.
01.02Custom fields disappear from workflow action dropdowns3x recurring

We hit this three separate times: a custom field exists in Settings → Custom Fields, but doesn't surface in the workflow builder's action dropdown when trying to update it. Field looks invisible to automations.

Fix: Confirm field type matches what the action expects (text vs dropdown vs date). If still missing, recreate the field with a clean snake_case name, or hit it via API instead of UI action.
01.03First name / last name forced as separate fields in GHLForm UX

GHL splits name into two fields by default in checkout and contact forms. UX research keeps showing a single "Full name" field converts better, especially on mobile.

Fix: Collect "Full name" on a custom overlay form. Before submit, split on the first space, write to first_name and last_name hidden inputs on the underlying GHL form, then trigger submit. Pattern in Section 5.
01.04Checkout order summary bloats page heightMobile fold

The default order summary block shows every line item (Workshop, Ticket, Offer, Subtotal, Tax). On mobile, this pushes the CTA below the fold and adds 20–30% page height.

Fix: Hide line items via CSS, show only the total. When the VIP checkbox toggles, just update the total figure — don't reveal more rows. 
/* hide bloated line items, keep total */
.order-summary .line-item { display: none !important; }
.order-summary .total-row { display: flex !important; }

02 ·CSS Collisions with GHL Wrapper Styles

GHL ships a default stylesheet that wraps everything in .c-section and .hl_page-creator. Custom CSS loses specificity fights unless you scope it.

02.01Looping button animations look gimmicky on premium audiencesTrust signal

The first round of niche LPs shipped with infinite nudge-glow and nudge-shimmer keyframes on the gold CTA. Professional audiences (attorneys, therapists) read these as "ad-y" and stall on the page. Static buttons + a hover state convert better.

Fix: Delete every @keyframes nudge-* block and the animation: property on .gold-btn. Keep only the hover transition (color shift + box-shadow).
02.02GHL's global button styles override your custom buttonsSpecificity

If your HTML block uses bare button { } or .btn { } selectors, GHL's global page styles will outweigh them and your button looks half-styled.

Fix: Scope every custom button selector to a parent class unique to your block. Example: .cod-hero .gold-btn beats .btn every time without needing !important.
02.03100% width inside HTML blocks isn't actually full viewportLayout

GHL wraps each HTML block in a container with horizontal padding. width: 100% respects that container, so full-bleed sections appear inset.

Fix: For full-bleed sections, use the negative-margin trick: 
.cod-hero {
  width: 100vw;
  position: relative;
  left: 50%;
  right: 50%;
  margin-left: -50vw;
  margin-right: -50vw;
}
02.04Inline critical CSS, defer the restPattern

GHL's own stylesheet loads first, then your HTML block styles. Above-the-fold styles can render late, causing a flash of unstyled content.

Fix: Put critical hero CSS inline inside <style> at the top of the HTML block. Move below-fold rules into a deferred stylesheet or section-specific blocks.

03 ·Custom Fields & Merge Field Gotchas

Naming conventions and trigger filters that look optional but break automations silently.

03.01"Pipeline Stage Changed" triggers fire on every pipelineSilent over-fire

The trigger has no pipeline scope by default. If a contact moves through any pipeline (not just Sales), the workflow fires. We had emails going out on AEvent stage shifts because of this.

Fix: Add the "Pipeline" filter inside the trigger config and lock it to Sales Pipeline only. Test by moving a dummy contact through a non-Sales pipeline and confirming nothing fires.
03.02Merge fields are strict on field ID, not display nameNaming

If you rename a custom field's display label but it was referenced in workflow templates or emails by the old machine name, the merge silently outputs empty string.

Fix: Use snake_case field names from day one (appointment_status, not "Appointment Status"). Don't rename fields mid-flight — clone and migrate.

04 ·Images & Asset Hosting

This is the section that directly applies to Michelle's build. We're moving off GitHub-hosted images for COD funnels.

04.01GitHub-hosted images are a fragile dependencyToday's decision

The HTML for Michelle currently references images from a GitHub repo. Each page load pulls them over the public internet. If the repo goes private, the account closes, or GitHub rate-limits anonymous traffic, every funnel page breaks images simultaneously. Original source for those images is Google Drive — COD already has the assets.

Fix: Upload each image into GHL Media before publish. Replace every raw.githubusercontent.com/... URL in the HTML with the GHL Media URL. Same applies for any logo, headshot, or testimonial photo.
04.02GHL Media works fine for small assets, struggles with galleriesWhen to bail

For under ~20 images, GHL Media is the right call. For full photo galleries or assets that change weekly, GHL Media gets noisy fast.

Fix: Single-purpose funnels (Michelle): GHL Media. Volume builds or anything CMS-shaped: external CDN (Cloudflare R2, Vercel Blob) with a stable URL pattern.
04.03loading="lazy" sometimes gets stripped on saveAfter publish

GHL's HTML sanitiser has been spotted stripping loading="lazy" off <img> tags during save in some templates. Above-the-fold images end up loading eagerly anyway, but hero images you wanted lazy can disappear from the attribute set.

Fix: Inspect the published page (not the editor preview) and confirm the attribute survived. If stripped, swap to native intersection-observer loading in JS or accept eager load.

05 ·Forms, Overlays & the Hidden GHL Form Pattern

The cleanest way to ship custom multi-step forms without losing GHL's CRM tagging, workflow triggers, or pipeline automations.

05.01The hidden-GHL-form patternCanonical approach

Build the custom form as the visible UI. Keep the native GHL form on the page but hidden. On submit of the custom form, populate the hidden GHL form's inputs by name attribute, then call .submit() on it. Everything downstream (workflows, pipeline, contact creation) fires normally because GHL sees a real form submit.

Fix: 
<style>
  .ghl-hidden-form { position: absolute; left: -9999px; opacity: 0; }
</style>

<form id="custom-form"> ... </form>
<div class="ghl-hidden-form">
  <!-- paste the native GHL form embed here -->
</div>

<script>
  document.querySelector('#custom-form').addEventListener('submit', function(e) {
    e.preventDefault();
    const ghl = document.querySelector('.ghl-hidden-form form');
    ghl.querySelector('[name="first_name"]').value = first;
    ghl.querySelector('[name="last_name"]').value  = last;
    ghl.querySelector('[name="email"]').value      = email;
    ghl.submit();
  });
</script>
05.02URL query params get stripped on GHL form submitAttribution loss

When the custom form submits and GHL redirects to its thank-you URL, query string params (utm_*, fbclid, gclid, custom values like ?email=...&phone=...) are dropped unless you forward them explicitly.

Fix: Before calling ghl.submit(), append the current window.location.search string to ghl.action: 
ghl.action = ghl.action + window.location.search;
ghl.submit();
05.03Form shrinks below fold on mobile after option selectMobile UX

Multi-step forms transition height when the user picks an option on Step 1. On phones, the form moves up and the new question lands below the viewport. People think the form broke and bounce.

Fix: After each step transition, scroll the form back into view: 
setTimeout(() => {
  form.scrollIntoView({ behavior: 'smooth', block: 'center' });
}, 100);
Test on a real iPhone and a real Android, not DevTools.
05.04Field name attributes must match GHL's exact expectationsMap carefully

If the GHL form expects email and you write user_email in the hidden form, the contact gets created without an email. CRM looks half-broken and the workflow's "send email" step silently no-ops.

Fix: Open the live native GHL form, inspect element on every input, copy the name attribute verbatim into the hidden form. Test field-by-field with a dummy submit and verify the contact lands with every field populated.

06 ·Tracking, Pixels & Head/Body Code

Where the pixels go and why placement order matters for PostHog, Hyros, and Meta CAPI.

06.01PostHog needs identify() after form submit for attributionA/B parity

Without an identify(email) call, PostHog tracks the user as anonymous before the form and as a different anonymous after. Splits get noisy because the same human counts as two people.

Fix: Call posthog.identify(email) as soon as you capture an email in the form (Step 1 or earlier). Confirm in PostHog Persons that the anonymous → identified merge happened.
06.02Tracking code goes in Settings → Tracking, not inline HTML blocksPlacement

If you paste GTM, PostHog, or Meta pixel snippets inside a custom HTML block, they fire whenever that block renders — which can be after other blocks or after JS execution. Order breaks attribution.

Fix: Put pixel/tag snippets in GHL Settings → Tracking (head + body slots). Reserve HTML blocks for content + form logic only.
06.03PostHog + Hyros must both be live on every funnel pageCross-check

If a funnel is partly built in the builder and partly in HTML blocks, it's easy to ship a page that has the Meta pixel but not Hyros, or PostHog but not the Hyros server-side ping. Attribution silently goes one-sided.

Fix: Pre-publish check on every page: PostHog Inspector shows pageview + identify, Hyros debug shows server hit, Meta Pixel Helper shows PageView. Three confirmations or don't ship.

07 ·Mobile & Responsive Issues

~90% of COD traffic is mobile. DevTools emulation lies. Real devices catch what emulation misses.

07.01DevTools mobile emulation is not enoughAlways test real

iOS Safari, Android Chrome, and in-app browsers (Instagram, Facebook, TikTok) all behave differently on viewport units, autoplay policy, and form keyboard interactions. We've shipped pages that looked perfect in DevTools and broke for half of paid traffic.

Fix: Real iPhone + real Android end-to-end test before publish. Walk the entire flow including a $1 test charge (refund after). Screenshot every step.
07.02Hero images often ignore max-width: 100% in HTML blocksOverflow

Inside GHL's container, raw <img> tags can still overflow on small screens, especially if the image's intrinsic width is huge.

Fix: Wrap hero images in a class with explicit responsive rules: 
.hero img { max-width: 100%; height: auto; display: block; }
Use a separate mobile-optimized image source via <picture> when the asset is heavy.
07.03Checkout breaks differently on iOS Safari vs Chrome AndroidPer-OS

iOS Safari has quirks with autofill on credit-card fields. Android Chrome can mis-render the Stripe iframe inside GHL checkout if there's a parent transform property anywhere above it.

Fix: Audit parent containers for transform, filter, or perspective — strip them off the checkout's ancestors. iOS test the autofill flow explicitly with a real saved card.

08 ·Video Embeds (Wistia, YouTube)

Wistia is the COD standard. Two config patterns — one for background hero, one for inline clickable. Don't mix them.

mutedtruefalseMobile browsers block unmute on autoplay; inline videos need sound on click
autoPlaytruefalseHero plays silently; inline waits for click
silentAutoPlaytrue—Forces silent autoplay where browser allows
controlsVisibleOnLoadfalsetrueHero is decorative; inline needs scrubber visible

08.01GHL preview shows the video wrong; live URL is the truthAlways publish-test

Wistia embed params (especially autoplay variants) frequently render incorrectly inside GHL's editor preview. The same embed works perfectly on the published page.

Fix: Never sign off a video from the preview. Publish to a staging slug, open the live URL on desktop and mobile, then confirm behavior.
08.02Click-to-timestamp navigation isn't supported in GHLScope cut

Designs that want "click chapter 3 to jump to 4:32 in the video" don't work cleanly inside GHL. The custom JS to scrub a Wistia or YouTube player is fragile and breaks on mobile.

Fix: Drop the feature from any GHL-hosted design at brief stage. If timestamps matter for a build, that build belongs on a hard-coded site, not GHL.

09 ·Performance & Page Speed

GHL is heavy by default. Custom HTML blocks add to that. Discipline matters more than tricks.

09.01Custom JS bloat compounds the GHL baselineDiscipline

GHL already injects ~400KB of base JS for the page builder runtime. Every HTML block you add layers more. Three blocks with their own jQuery copies = a slow LCP.

Fix: One <script> per page if possible. No jQuery unless absolutely needed. Defer non-critical scripts with async or defer.
09.02Font loading order causes FOUTHero pop-in

Custom Google Fonts loaded inside an HTML block render after GHL's defaults, so the hero text "pops" mid-fold.

Fix: Preload critical fonts: 
<link rel="preload"
      href="https://fonts.gstatic.com/.../font.woff2"
      as="font" type="font/woff2" crossorigin>
Use font-display: swap in the @font-face rule so text is always readable.
09.03Test Core Web Vitals on every meaningful changeDiscipline

Mobile LCP target: under 2.5s. CLS under 0.1. We've shipped niche LPs at 4s+ LCP because of unsized hero images and webfont swap.

Fix: Run PageSpeed Insights on the live published URL after each build. Treat anything above 3.0s LCP as a defect, not a feature request.

10 ·Builder ↔ HTML Parity

When you have to round-trip between the page builder and raw HTML, things drift. This is the section to read before deciding which approach to use for a given build.

10.01HTML approach loses template propagationTradeoff

Pages built in the standard template benefit from "edit master, propagate to all clients." Pages built as raw HTML blocks don't. A change to the master template doesn't reach Michelle's HTML pages automatically.

Fix: Keep one or the other per client. Don't mix. For Michelle (MA, low edit volume), HTML is the call. For everyone else, standard template remains. If a master-template change should reach Michelle, port it manually.
10.02Builder rebuilds drift across niche pagesConsistency

When the same page gets rebuilt for therapist, attorney, coaches, and educators in the page builder, small things diverge — button animation present on one but not another, footer link target attribute set on one but not another, FAQ accordion behaving differently.

Fix: Build one canonical page (therapist was the canonical for niche LPs). Lock it. Apply changes to canonical first, get sign-off, then clone to every other niche. Run the consistency checklist in Section 11 after each rebuild round.
10.03Nothing goes live without a screenshot or Loom walkthroughHard rule

Every COD page that ships gets a screenshot or short Loom before publish, sent to Kas in Slack for sign-off. No exceptions. This catches 80% of the issues this guide covers.

Fix: Build → screenshot or Loom every section → message Kas → wait for approval → publish. If it's urgent, say it's urgent in the message; don't skip the step.

11 ·Pre-Publish Checklist

Run this every time before a COD funnel page goes live. Takes about ten minutes if everything is right.

Before Publish

  • All images uploaded to GHL Media (no raw.githubusercontent.com URLs in HTML)
  • Logos and headshots cross-checked against the Drive folder COD already maintains
  • Hidden GHL form field names match the live native form exactly (test submit lands a full contact)
  • URL query params forwarded on form submit (action += window.location.search)
  • PostHog identify(email) called after first email capture
  • Pixels live in Settings → Tracking, not inline HTML blocks
  • PostHog Inspector + Hyros debug + Meta Pixel Helper all green on the live URL
  • No @keyframes nudge-* animations on CTA buttons
  • Custom button selectors scoped to a parent class, no bare button { }
  • Order summary collapsed to total only (line items hidden)
  • No "Add to Calendar" buttons anywhere on the page
  • Footer legal links open in new tab (target="_blank" rel="noopener")
  • Wistia params: hero = silent autoplay + loop; inline = muted false + controls visible
  • Real iPhone test: hero loads, form scrolls into view after option select, checkout works
  • Real Android test: same flow
  • PageSpeed Insights mobile LCP under 3.0s on the live URL
  • Screenshot or Loom sent to Kas for sign-off before publish
