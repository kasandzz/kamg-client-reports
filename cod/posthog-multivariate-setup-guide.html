PostHog Multivariate Testing Setup for COD Landing Pages

How to wire feature flags, experiments, and trustworthy variant tracking across every COD reg page. Keeps PostHog in GTM — no tag migration, no disruption. Adds the flag layer that's currently missing. Written for Rehan.

Owner: Rehan Reviewer: Kas Pages in scope: all COD landing pages (therapist, attorney, educators, coaches, main reg, future niches) Est. effort: 4–6 hours one-time + ~15 min per new test

TL;DR

The identify call is already live and working on /hto-workshop-therapist-counselor-b. PostHog is being loaded via GTM (container GTM-M677PMW). Identify fires correctly on form submit and on URL ?email= param. Good foundation.

But the pages have zero feature flag consumption code, meaning we cannot currently run PostHog experiments or multivariate tests. This brief fixes that in five phases — WITHOUT moving PostHog out of GTM:

  1. Phase 1 — Raise GTM tag priority + add anti-flicker CSS. PostHog stays in GTM. Tag fires on Consent Init with priority 1000. A tiny CSS rule hides test slots until flags resolve. No FOOC, no migration.
  2. Phase 2 — Standardize the identify pattern across every page using one shared snippet (cod-posthog.js).
  3. Phase 3 — Add the feature flag consumption pattern (onFeatureFlags + variant switch + exposure event).
  4. Phase 4 — Run a smoke test using a dummy 50/50 flag to confirm clean bucketing before any real test launches.
  5. Phase 5 — Roll out page by page with a QA checklist per page.

02Where Should PostHog Init Live?

The honest comparison of three options. We recommend keeping PostHog in GTM and fixing flicker at the CSS layer — the same pattern Optimizely and VWO use.

Option A — Keep in GTM, do nothing

Pros: zero disruption, consent mode keeps working, one source of truth for all tags, Akari/team already knows GTM.

Cons: you still have no flag consumption code, so you cannot run PostHog experiments at all. This isn't really an option — it's the status quo we're trying to exit.

Option B — Move to blocking head

Pros: zero flicker, fastest possible variant paint, config is code-reviewable.

Cons: you lose GTM-level control of PostHog. Consent mode has to be manually wired. Two places to manage PostHog config. Must pause the GTM tag cleanly or risk double-init. The team has to learn "PostHog lives in page source now."

Verdict: technically superior on paint speed, but the disruption isn't worth it for COD's current scale.

Option C — Keep in GTM, fix flicker with CSS (RECOMMENDED)

The industry-standard pattern used by Optimizely, VWO, Convert, and AB Tasty. Hide the test-slot elements with CSS until flags resolve, then reveal them. Safety timeout prevents users from ever seeing a permanently blank page if PostHog fails.

<style>
  /* Hide elements under test until PostHog applies the variant */
  [data-test-slot] { visibility: hidden; }
  html.ph-ready [data-test-slot] { visibility: visible; }
</style>
<script>
  /* Safety timeout — reveal after 1500ms even if PostHog never loads (adblockers, network failure) */
  setTimeout(function() { document.documentElement.classList.add('ph-ready'); }, 1500);
</script>

The onFeatureFlags callback adds html.ph-ready as soon as it runs. Normal load: ~200–500ms blank hero, then variant reveals. Adblocker/failure: 1500ms blank, then control reveals. Never a flash of wrong content.

Pros: keep GTM, zero FOOC, consent mode works, same mental model as today, battle-tested pattern, fallback for adblockers.

Cons: blank hero for up to 500ms on slow mobile. Measured LCP impact is negligible in real conditions. Adblocker users never see variants (they're excluded from the test anyway, so this is actually correct behavior).

Decision

Go with Option C. PostHog stays in GTM. We add anti-flicker CSS + the flag consumption pattern + cod-posthog.js identify helper. Rehan does zero GTM tag migration — he only raises the existing tag's firing priority and adds two small blocks to each page's HTML. Everything else (FB Pixel, Google Ads tags, consent mode) keeps running exactly as today.

03What's Actually New vs Today

The location of PostHog init is a secondary question. The real changes — and the real value — are these four things you don't have today.

1. Feature flag consumption code

Right now, no COD page reads feature flags. No getFeatureFlag, no onFeatureFlags, no variant branching. This is the #1 blocker — you literally cannot run a PostHog experiment today even if you created the flag in PostHog's UI. Nothing on the page would react to it. Adding the EXPERIMENTS map pattern (Phase 3) fixes this.

2. Exposure events

PostHog's stats engine needs to know who saw which variant at what moment. That's what the $feature_flag_called event does — it's PostHog's reserved exposure event. Without it, the experiment results UI shows zero exposures and you can't tie variant to downstream conversion (ticket purchase, enrollment). The EXPERIMENTS loop fires this automatically.

3. Super properties on every event

Today every event from the reg page is untagged — no niche, no device_category, no page_template. That means you can't segment results by "mobile only" or "therapist niche only" or "paid social only" without doing manual joins. Adding posthog.register() in the init loaded callback attaches these properties to every subsequent event automatically — so segment breakdowns work in PostHog's UI with one click instead of writing SQL against the warehouse.

4. The data-test-slot HTML contract

Instead of targeting elements by fragile CSS selectors (.hero h1), every swappable element gets a stable data-test-slot="hero-headline" attribute. One-time cost to tag elements, but future tests are decoupled from CSS refactors. When the designer rebuilds the hero next month, every active experiment keeps working.

Bonus: anti-flicker CSS

The [data-test-slot] { visibility: hidden } rule is what makes Option C work. It's not strictly "new tracking" — it's the missing glue that lets you run PostHog experiments through GTM without FOOC.

04Drawbacks of the Whole Approach

Honest list. Every one of these is a real tradeoff. Go in with eyes open.

Complexity tax on every test

Every new test now requires (1) create flag in PostHog, (2) add entry to EXPERIMENTS map in page code, (3) mark target elements with data-test-slot, (4) deploy page change, (5) validate in QA, (6) launch experiment in PostHog. That's 6 steps instead of "edit the page and hope." Worth it for statistical trust. Painful for 5-minute spot tests.

PostHog MTU billing

PostHog charges per monthly tracked user. We set person_profiles: 'identified_only' to avoid creating profiles for anons, but every identified lead still counts. At COD's current lead volume this is manageable; if you ever 10x traffic, check pricing. Session recordings are the bigger cost — consider downsampling to 10% of sessions after rollout if quota runs hot.

Adblocker blindspot

Roughly 10% of web traffic blocks PostHog entirely. Those users see the control variant (thanks to our safety timeout) and aren't counted in experiment results. This is actually fine for relative lift measurement — PostHog's stats engine doesn't need to see 100% of users — but it means absolute traffic counts in PostHog will be ~10% lower than Meta/Hyros. Don't use PostHog as your source of truth for total traffic.

Brief blank hero on slow connections

The anti-flicker CSS hides test-slot elements until flags resolve. On slow 3G / poor connections, that's ~500ms of blank space where the hero headline would be. Measured LCP impact in real conditions: small (~100ms median). But it's a visible tradeoff — the hero appears to "pop in" rather than being there on initial paint. If this ever becomes a perceived-performance problem, we can downscope which elements get the hidden treatment (e.g. only headline, not CTA).

Mental discipline required

Stop rules, minimum sample sizes, not peeking early, calculating MDE before launching. If the team ships based on gut after 48 hours of data, PostHog is no better than GHL's split test. The tooling only matters if the process respects it. Every experiment needs a pre-declared hypothesis, primary metric, MDE, stop rules, and minimum runtime — otherwise you're back to guessing.

One experiment at a time per page (recommended)

Technically PostHog supports multiple overlapping experiments on one page. In practice, running two tests on the same reg page contaminates both — you can't tell if the lift came from headline test or CTA test. Default to one active experiment per page; if you must run two, make sure they target completely different elements and disclose the interaction in the analysis.

Rehan becomes a single point of dependency

Every new test requires a Rehan deploy. That's fine for now but means test velocity is capped by his bandwidth. Medium-term fix: give the team access to PostHog Web Experiments (visual editor) for copy-only tests, reserve code-based experiments for structural changes. Only turn on Web Experiments after Phase 1–5 are solid.

05Target Architecture

One shared script, one init call, one flag pattern, used by every page in the COD family.

// Load order on every COD landing page

<head>
  1. Anti-flicker CSS           // [data-test-slot] { visibility: hidden }
  2. Safety-timeout script      // reveal after 1500ms failsafe
  3. EXPERIMENTS map + onFeatureFlags // queue the variant handler
  4. GTM snippet                // loads PostHog + FB Pixel + Google Ads as before
  5. Other head tags            // fonts, meta, etc
</head>

<body>
  6. Page content with data-test-slot markers
  7. GTM fires PostHog Init tag // priority 1000, Consent Init trigger
  8. posthog.register()          // super props via the loaded callback
  9. posthog.onFeatureFlags()     // variant applies, html.ph-ready added
  10. Test slots reveal         // CSS visibility flips on
  11. cod-posthog.js             // shared identify + form wiring (async)
  12. Form submit → identify     // email becomes distinct_id
</body>
PostHog stays in GTM

The entire architecture is built around keeping PostHog where it is today — inside the GTM container, alongside FB Pixel, Google Ads, and consent mode. No tag migration, no double-init risk, no team retraining. The only GTM change is bumping the PostHog Init tag's firing priority so it loads first among GTM tags.

06Phase 1 — GTM Tag Priority + Anti-Flicker CSS

Two small changes to make PostHog fast enough for experiments without leaving GTM.

Step 1 — Raise the PostHog GTM tag firing priority

In the GTM container (GTM-M677PMW):

  1. Open the existing PostHog Init tag (search container for posthog.init if you're not sure which one it is).
  2. Open Advanced Settings → Tag firing priority. Set to 1000. (Higher = fires first among GTM tags.)
  3. In the tag's triggers, change the trigger to Consent Initialization — All Pages. This is GTM's earliest possible trigger — fires before Page View, before DOM Ready. Works alongside your existing consent mode setup.
  4. Save → Submit → Publish.

Result: PostHog is now the first thing GTM bootstraps. Typical load time drops from 400–800ms post-paint to 100–300ms post-paint. Not instant, but fast enough to pair with the CSS approach in Step 2.

Step 2 — Verify the PostHog init config in GTM

Inside the GTM tag's Custom HTML body, the posthog.init call needs these settings. Update the tag if any are missing:

<script>
posthog.init('phc_REPLACE_WITH_COD_PROJECT_KEY', {
  api_host: 'https://us.i.posthog.com',
  person_profiles: 'identified_only',      // saves MTU cost
  capture_pageview: true,
  capture_pageleave: true,
  autocapture: false,                      // explicit events only
  advanced_disable_feature_flags: false,   // CRITICAL — must be false for experiments
  advanced_disable_decide: false,          // CRITICAL — /decide must run
  loaded: function(ph) {
    // Super properties — attached to every subsequent event
    ph.register({
      niche: document.body.dataset.niche || 'unknown',
      page_template: document.body.dataset.template || 'reg',
      page_path: location.pathname,
      device_category: window.innerWidth < 768 ? 'mobile' : 'desktop'
    });
    // Mark document ready so anti-flicker CSS reveals test slots
    document.documentElement.classList.add('ph-ready');
  }
});
</script>
Why each flag matters
  • person_profiles: 'identified_only' — PostHog charges per MTU. Anonymous profiles eat quota.
  • autocapture: false — we capture explicit events only, no accidental PII capture from form inputs.
  • advanced_disable_feature_flags: false — default is false, but be explicit. If this flips, experiments break silently.
  • advanced_disable_decide: false — the /decide call is what actually resolves flag values. Must run.
  • html.ph-ready class add — this is the anti-flicker signal. CSS in Step 3 uses it to reveal test-slot elements only after PostHog has loaded and flags are resolved.

Step 3 — Anti-flicker CSS + safety timeout in page <head>

Add this small block to the very top of <head> on every COD landing page. Before fonts, before CSS, before GTM. It's ~300 bytes inline — no performance cost.

<!-- Anti-flicker: hide variant-controlled elements until PostHog applies the variant -->
<style>
  [data-test-slot] { visibility: hidden; }
  html.ph-ready [data-test-slot] { visibility: visible; }
</style>
<script>
  /* Safety timeout: reveal after 1500ms even if PostHog never loads (adblockers, network failure) */
  setTimeout(function() {
    document.documentElement.classList.add('ph-ready');
  }, 1500);
</script>

How it works end to end

  1. User loads page → browser paints everything EXCEPT elements marked data-test-slot (they're visibility: hidden).
  2. GTM bootstraps → PostHog Init tag fires first (priority 1000, Consent Init trigger) → posthog.init runs.
  3. PostHog loaded callback runs → super props registered → html.ph-ready class added.
  4. CSS sees html.ph-ready → test slots become visible with the variant already applied (Phase 3 code runs before the class is added).
  5. If PostHog never loads (adblocker, network failure), the 1500ms safety setTimeout adds ph-ready anyway → user sees the control variant → nothing stays blank forever.
What this achieves

Zero disruption to GTM. PostHog stays inside the container. FB Pixel, Google Ads, consent mode — all untouched. Only net changes: (1) one tag priority bump, (2) ~300 bytes of inline CSS/JS in page head, (3) the ph-ready class add in the PostHog loaded callback. Rehan's GTM edit takes ~10 minutes. The per-page changes take ~5 minutes each.

Get the PostHog project key (if the current GTM tag doesn't already have it set)

The key starts with phc_. Log into PostHog → Settings → Project → API Key. If the GTM tag already has a key in it, confirm it matches the production PostHog project. Don't commit the key to a public repo.

07Phase 2 — Standardized Identify Pattern

One file, one import, one helper. Stop copy-pasting identify code per page.

Create the shared script

Host this at a stable URL (GHL media library, or a CDN like GitHub Pages). Reference it from every COD landing page.

/* cod-posthog.js · v1.0.0 · shared identify + event helpers for COD landing pages */
(function() {
  'use strict';

  // Wait for posthog to exist (init is in head, so this is usually instant)
  function whenReady(cb) {
    if (window.posthog && typeof window.posthog.identify === 'function') return cb();
    let tries = 0;
    const poll = setInterval(function() {
      if (window.posthog && typeof window.posthog.identify === 'function') {
        clearInterval(poll);
        cb();
      } else if (++tries > 50) {
        clearInterval(poll);
        console.warn('[cod-posthog] posthog never loaded after 5s');
      }
    }, 100);
  }

  // Normalize and validate an email before identifying
  function cleanEmail(raw) {
    if (!raw || typeof raw !== 'string') return null;
    const e = raw.toLowerCase().trim();
    return e.includes('@') && e.includes('.') ? e : null;
  }

  // Identify a user — call this on form submit, URL email param, or any known contact point
  window.CODIdentify = function(email, extras) {
    const clean = cleanEmail(email);
    if (!clean) return false;
    whenReady(function() {
      window.posthog.identify(clean, Object.assign({
        email: clean,
        source: location.pathname,
        first_touch_at: new Date().toISOString(),
        niche: document.body.dataset.niche || 'unknown'
      }, extras || {}));
      console.log('[cod-posthog] identified', clean);
    });
    return true;
  };

  // Standard capture wrapper — enforces consistent event naming
  window.CODCapture = function(event, props) {
    whenReady(function() {
      window.posthog.capture(event, props || {});
    });
  };

  // Auto-wire form submit on any form tagged [data-cod-lead-form]
  function wireForms() {
    const forms = document.querySelectorAll('form[data-cod-lead-form], #_builder-form');
    forms.forEach(function(form) {
      if (form.dataset.phAttached) return;
      form.dataset.phAttached = 'true';
      form.addEventListener('submit', function() {
        const emailInput = form.querySelector('input[type="email"], input[name*="email" i]');
        const nameInput = form.querySelector('input[name*="name" i], input[name*="full_name" i]');
        const email = emailInput ? emailInput.value : null;
        const name = nameInput ? nameInput.value : null;
        if (email) {
          window.CODIdentify(email, { full_name: name, stage: 'lead' });
          window.CODCapture('lead_registered', {
            full_name: name,
            form_id: form.id || form.className.split(' ')[0]
          });
        }
      });
    });
  }

  // Auto-wire URL ?email= param for nurture link returns
  function wireUrlEmail() {
    const urlEmail = new URLSearchParams(location.search).get('email');
    if (urlEmail) window.CODIdentify(urlEmail, { source: 'url-param' });
  }

  if (document.readyState === 'loading') {
    document.addEventListener('DOMContentLoaded', function() {
      wireForms();
      wireUrlEmail();
    });
  } else {
    wireForms();
    wireUrlEmail();
  }
})();

How pages load it

After the head init script, anywhere in body (async is fine):

<body data-niche="therapist" data-template="reg-workshop">
  ...
  <script src="https://your-cdn/cod-posthog.js" async></script>
</body>
Win

Once this is in place, every new COD landing page only needs two things: (1) the head init script, (2) this script tag. Identify and form tracking are automatic. No more copy-pasting inline handlers per page.

08Phase 3 — Feature Flag Consumption Pattern

How to actually apply a variant to the DOM and report exposure back to PostHog.

The pattern

Add this to any page that has an active test. Nothing happens on pages without a test — the callback just no-ops.

<script>
(function() {
  // Map of flag_key -> variant handlers
  // Add one entry per active experiment on this page
  const EXPERIMENTS = {
    'reg-therapist-hero-v1': {
      control: function() { /* do nothing, original DOM */ },
      'variant-a': function() {
        const h = document.querySelector('[data-test-slot="hero-headline"]');
        if (h) h.textContent = 'The exact system therapists use to book $5K clients without being salesy';
      },
      'variant-b': function() {
        const h = document.querySelector('[data-test-slot="hero-headline"]');
        if (h) h.textContent = 'How 200+ therapists filled their practice in 90 days (no ads)';
      }
    }
  };

  function applyExperiments() {
    Object.keys(EXPERIMENTS).forEach(function(flagKey) {
      const variant = window.posthog.getFeatureFlag(flagKey);
      if (!variant) return;
      const handler = EXPERIMENTS[flagKey][variant];
      if (typeof handler === 'function') handler();

      // Fire the exposure event so PostHog can attribute downstream conversions
      window.posthog.capture('$feature_flag_called', {
        '$feature_flag': flagKey,
        '$feature_flag_response': variant
      });
    });
  }

  // Run as soon as flags are ready — this resolves BEFORE first paint when init is in head
  if (window.posthog && window.posthog.onFeatureFlags) {
    window.posthog.onFeatureFlags(applyExperiments);
  } else {
    // Fallback: queue it for when posthog loads
    (window.posthog = window.posthog || []).push(['onFeatureFlags', applyExperiments]);
  }
})();
</script>

The HTML contract

Mark any element that might be swapped by a variant with a data-test-slot attribute. This keeps variant code decoupled from fragile CSS selectors.

<h1 data-test-slot="hero-headline">
  Original headline that control sees
</h1>

<p data-test-slot="hero-subhead">
  Original subhead
</p>

<button data-test-slot="hero-cta">
  Register now
</button>
Why data-test-slot over class selectors

If we target .hero h1, a future CSS refactor breaks every active experiment silently. data-test-slot is a contract — it survives design changes, is searchable, and makes experiment code self-documenting.

Flag lifecycle

  • Running — test is live, traffic is splitting, nobody touches the code
  • Concluded — in PostHog, mark experiment complete. Note the winner in the experiment description.
  • Implemented — the winning variant becomes the default DOM. Flag key stays in code but with all handlers pointing to the new default. Feature flag is set to 100% variant-X in PostHog.
  • Archived — 30 days later, delete the flag from PostHog AND remove the handler from EXPERIMENTS. Keeps the experiment file clean.

10Standard Events

These must fire identically across every COD landing page. The BigQuery warehouse joins on exact event names.

Do NOT capture these

  • Anonymous browsing events with autocapture — we turned autocapture off in Phase 1, keep it off.

11Phase 4 — Smoke Test Before Any Real Test

Run this before trusting the pipeline with a revenue-affecting test. Cost: one afternoon.

Create the smoke test flag

  1. PostHog → Feature Flags → New feature flag
  2. Key: infra-smoke-test
  3. Type: Multiple variants → control (50%), variant-a (50%)
  4. Release conditions: 100% of users

Wire it into one page

Add a harmless DOM change to the therapist reg page so you can visually confirm which bucket you're in:

const EXPERIMENTS = {
  'infra-smoke-test': {
    control: function() {
      document.title = '[A] ' + document.title;
    },
    'variant-a': function() {
      document.title = '[B] ' + document.title;
    }
  }
};

Validation checks (all must pass)

  • Incognito tab 1: load page → check tab title has [A] or [B] prefix → no flicker → submit form → PostHog Persons shows email distinct_id.
  • Incognito tab 2: repeat. Different browser. Confirm you get a different bucket at least once over ~10 incognito sessions.
  • PostHog → Events → Live: confirm $pageview fires BEFORE $feature_flag_called, and both fire before lead_registered.
  • PostHog → Persons: after 20 sessions, verify the infra-smoke-test property on each person is set and roughly 50/50 distribution.
  • Network tab: /decide/ request returns before any DOM paint containing the variant. If the request is slow (>200ms), flags fall back to bootstrap/cached.
  • PostHog → Session recordings: open one recording. Confirm no flicker on hero headline swap during load.
Pass criteria

All six checks green → proceed to Phase 5 rollout. If ANY check fails, do not run real experiments. Debug first. The most common failure is /decide firing too late — fix by ensuring the PostHog snippet is the first script in <head>.

14Common Pitfalls

Every one of these has bitten a team doing PostHog experiments. Don't learn them the hard way.

Flicker / flash of original content (FOOC)

Symptom: users see the control variant flash briefly before the test variant paints. Cause: PostHog's /decide endpoint hadn't returned yet when the page painted. Fix: the anti-flicker CSS in Phase 1 Step 3 hides test-slot elements with visibility: hidden until the html.ph-ready class is added. PostHog's loaded callback adds the class once flags resolve. Safety timeout guarantees the page reveals after 1500ms even if PostHog never loads. This is the same pattern Optimizely, VWO, and Convert all use.

Identify called before init

Symptom: identify silently drops, distinct_id stays anonymous, PostHog persons fill with unlinked UUIDs. Cause: form submit handler runs before window.posthog exists (e.g. user submits in the first 200ms). Fix: whenReady helper in cod-posthog.js polls for 5 seconds before giving up. Good enough for 99.9% of real-world submits.

Using alias instead of identify

Symptom: anonymous and identified profiles never merge, conversions look orphaned. Cause: someone copy-pasted a snippet from an older PostHog doc. Fix: always use identify. alias is legacy and not needed for this setup.

Changing flag variants mid-test

Symptom: users who joined variant-a in week 1 suddenly see variant-b in week 2, conversions look random. Cause: edit to the flag's rollout % or variant definition after traffic started. Fix: once a test is live, the flag is frozen. If you need to change it, kill the experiment and start a new flag with -v2 suffix.

Running GHL and PostHog tests on the same page

Symptom: bucket contamination, inconsistent results, unexplainable lifts/drops. Cause: both systems trying to rewrite the DOM. Fix: pick one. PostHog for real tests, GHL only for emergencies.

Forgetting to fire the exposure event

Symptom: PostHog experiments view shows zero exposures even though people are obviously seeing variants. Cause: calling getFeatureFlag() does NOT auto-fire $feature_flag_called in all client libs — you have to capture it explicitly. Fix: the Phase 3 pattern fires it inside the EXPERIMENTS loop. Don't remove that line.

Testing too small a change with too little traffic

Symptom: 6-week test returns "no significant result," team shrugs and moves on. Cause: MDE was bigger than the realistic effect size. Fix: before starting a test, use PostHog's built-in experiment calculator. If your MDE at current traffic is 20%+ and the change is a button color, don't run it. Test bigger swings instead.

Not waiting for PostHog's recommended sample size

Symptom: team declares winner after 3 days, ships it, conversion drops. Cause: peeking early with frequentist mindset. Fix: PostHog Bayesian mode is fine to peek on, BUT only ship when probability-to-beat-control ≥ 95% AND minimum sample size (usually 300+ conversions per variant) is hit. Not one without the other.

A. Page <head> block (full, ready to paste)

This is what goes in the page head on every COD landing page. PostHog itself continues to load from GTM — this block only handles the anti-flicker CSS and the experiment scaffolding.

<!-- 1. Anti-flicker CSS + safety timeout -->
<style>
  [data-test-slot] { visibility: hidden; }
  html.ph-ready [data-test-slot] { visibility: visible; }
</style>
<script>
  setTimeout(function() {
    document.documentElement.classList.add('ph-ready');
  }, 1500);
</script>

<!-- 2. Active experiments on this page (queues variant handler for when PostHog loads from GTM) -->
<script>
(function() {
  // Map of flag_key -> variant handlers. Add entries as tests launch.
  window.__codExperiments = {
    // 'reg-therapist-hero-v1': {
    //   control: function() {},
    //   'variant-a': function() {
    //     const h = document.querySelector('[data-test-slot="hero-headline"]');
    //     if (h) h.textContent = 'New headline text';
    //   }
    // }
  };

  function applyExperiments() {
    const EXP = window.__codExperiments || {};
    Object.keys(EXP).forEach(function(key) {
      const v = window.posthog.getFeatureFlag(key);
      if (!v || !EXP[key][v]) return;
      EXP[key][v]();
      window.posthog.capture('$feature_flag_called', {
        '$feature_flag': key,
        '$feature_flag_response': v
      });
    });
  }

  // Queue for when PostHog loads from GTM. If PostHog is already loaded (e.g. SPA nav), run immediately.
  function waitAndApply() {
    if (window.posthog && typeof window.posthog.onFeatureFlags === 'function') {
      window.posthog.onFeatureFlags(applyExperiments);
    } else {
      setTimeout(waitAndApply, 50);
    }
  }
  waitAndApply();
})();
</script>
