The 1-day filter, and why it exists

Someone who upgrades while the workshop is still fresh is emotionally the same as a bundled buyer — they're riding the momentum of the live event. Someone who upgrades a day later has done real consideration work: they watched, went away, thought about it, came back and paid more. That emotional state deserves different copy. So before anything else, the Master Routing Workflow checks days_since_workshop_attendance. If that value is 0, the contact is routed out of this system entirely (they belong to the existing bundled/same-day flow — check with Kas on the destination). If the value is ≥ 1, they continue into one of the two upgrade-later sequences.

The Thursday 11:59pm cutoff, and why it splits into two sequences

The VIP Friday Q&A is the anchor event of both sequences. If a contact upgrades Monday through Thursday, Friday is 0–4 days away — tight runway, no time for a weekend nurture. They get Sequence A: six emails, minimal filler, straight to Friday details and conversion (including a standalone replay delivery at +24h).

If a contact upgrades Friday through Sunday (or any time after Thursday 11:59pm client-TZ), Friday is 5–7 days away. That's a full week of potential silence before the anchor event. They get Sequence B: seven emails including a standalone replay delivery, a weekend value drop, and a merged mid-week prep/details email, designed to keep the relationship warm across the longer runway.

Sequence A Voice
Tight, action-oriented. Friday is around the corner. They already considered and committed — no extended nurture needed.
Sequence B Voice
Patient, coach-like. The week-long wait is framed as intentional — not a "sorry you have to wait" apology. Honors the considered decision, sets up the runway as useful.

Step 4 — If/Else Node B: Thursday cutoff branching

This is the core routing logic. You're comparing the contact's purchase_timestamp against Thursday 23:59 of the week containing that timestamp.

The cutoff formula (GHL math)

GHL doesn't have a single-line "is this before this week's Thursday 23:59" function, so build it as two helper fields then one comparison:

Cutoff Math — Pseudo-expression
// Step 4a — compute this_weeks_thursday_2359 (helper field)
// In GHL Custom Values or Math Op node:

this_weeks_thursday_2359 =
    DATE_TRUNC('week', purchase_timestamp, 'Monday')  // Monday 00:00 of the same week
    + INTERVAL '3 days'                               // + 3 days -> Thursday 00:00
    + INTERVAL '23 hours 59 minutes'                  // + 23:59 -> Thursday 23:59

// Step 4b — compare

IF purchase_timestamp <= this_weeks_thursday_2359
    THEN -> Sequence A branch
    ELSE -> Sequence B branch

// TIMEZONE NOTE:
// All dates above must evaluate in COD's client timezone (see Section 9 open question
// — confirm whether it's US/Eastern, US/Central, etc.). If GHL evaluates in UTC by
// default, you'll need to convert explicitly.

If GHL doesn't support DATE_TRUNC directly

Fallback — Day-of-week check
// Alternative using day-of-week (1=Mon, 7=Sun):

dow = DAYOFWEEK(purchase_timestamp)   // 1-7
hr  = HOUR(purchase_timestamp)        // 0-23
min = MINUTE(purchase_timestamp)      // 0-59

IF dow < 4
   OR (dow = 4 AND hr < 23)
   OR (dow = 4 AND hr = 23 AND min <= 59)
   THEN -> Sequence A   // Mon-Wed any time, or Thu up to 23:59
   ELSE -> Sequence B   // Fri/Sat/Sun, or Thu after 23:59 (which doesn't exist — but guard anyway)
