Why this stack (and what we explicitly rejected)

This was locked on 2026-04-24 to stop re-evaluating per client. The decisions below are not preferences — they are constraints. Don't re-litigate without bringing me in.

FRAMEWORK

Astro

Static-first, component reuse, team-friendly diffs, zero JS by default. Wins on Core Web Vitals which matters for ad Quality Score.

Astro 6.1+Next.jsRemixGatsby

HOSTING

Vercel

Preview URL per PR, auto-deploy from GitHub, custom domains, edge caching. Free tier covers most client sites.

VercelNetlify (fallback)AWS Amplify

ANALYTICS + EXPERIMENTS

PostHog (one vendor)

Events, A/B tests, feature flags, session replay, heatmaps, surveys. Same SDK, same dashboard, no five-vendor sprawl.

PostHogGrowthBookHotjarClarityMixpanel

PERF GATE

Lighthouse CI

GitHub Actions runs Lighthouse against the Vercel preview URL and blocks PRs that regress Core Web Vitals.

Lighthouse CISpeedCurveCalibre

STYLING

Tailwind v4 + tokens

CSS variables in tokens.css, Tailwind utilities for layout, scoped <style> blocks for component-specific.

Tailwind v4CSS variablesstyled-componentsSCSS

CMS (DEFAULT)

None (file-based)

Astro content collections in src/content/. We edit via Slack request loop. Add Sanity only when client wants self-edit.

Astro contentSanity (on demand)PayloadDecapStrapi

Why we said no to GrowthBook, Clarity, Hotjar

PostHog already does what they do. Five vendors means five SDKs, five dashboards, five contracts. PostHog is one SDK with feature flags, A/B, replay, heatmaps, surveys all in the same product. We pay for it once.

Why not Next.js

Next is great when you need server components, ISR, or a full app. Marketing microsites don't. Astro outputs static HTML with islands of interactivity only where we need them. Faster builds, faster pages, cheaper hosting, smaller cognitive load.

Astro anatomy — pages, components, layouts

Three concepts. That's the whole framework for our purposes. If you know JSX/Vue/Svelte you'll feel home in 20 minutes.

Pages = routes

Anything in src/pages/ becomes a route, file-based.

FileRoute
src/pages/index.astro/
src/pages/about.astro/about
src/pages/services/ductwork.astro/services/ductwork
src/pages/blog/[slug].astro/blog/:slug (dynamic)

Components = .astro files

// src/components/CTA.astro
---
interface Props {
  headline: string;
  buttonText: string;
  href: string;
}
const { headline, buttonText, href } = Astro.props;
---

<section class="cta">
  <h2>{headline}</h2>
  <a href={href} class="btn">{buttonText}</a>
</section>

<style>
  .cta { padding: 4rem 2rem; text-align: center; background: var(--color-bg-emphasis); }
  .btn { background: var(--color-cta); color: white; padding: 1rem 2rem; border-radius: 8px; }
</style>

Use a component in a page

---
import CTA from '../components/CTA.astro';
import BaseLayout from '../layouts/BaseLayout.astro';
---

<BaseLayout title="Redstone — Manufacturing">
  <CTA
    headline="Get a quote in 24 hours"
    buttonText="Request quote"
    href="/contact"
  />
</BaseLayout>

Layouts = page shells

// src/layouts/BaseLayout.astro
---
interface Props { title: string; description?: string; }
const { title, description } = Astro.props;
---
<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width" />
    <title>{title}</title>
    <link rel="stylesheet" href="/styles/global.css" />
  </head>
  <body>
    <slot />  <!-- page content injects here -->
  </body>
</html>

Content collections (copy-heavy pages)

When a page has a lot of text that a copywriter needs to edit without touching layout, use a content collection. Markdown frontmatter + body, type-checked at build time.

// src/content/services/ductwork.md
---
title: "Custom ductwork fabrication"
hero_headline: "Engineered ductwork in 5 days"
hero_subhead: "..."
cta_text: "Request a quote"
seo_description: "..."
---

## Capabilities

Long-form copy here that I/Akari can edit without breaking the layout.

This is our file-based CMS

Astro content collections + git = a CMS that costs nothing, has no admin UI to maintain, and gives us version control for free. For Redstone we use this exclusively. See section 13 for when to add Sanity instead.

2. Tailwind v4 (layout/spacing)

# add Tailwind to a client project
cd kamg-ops/sites/redstone/astro-project

Use Tailwind utilities for layout. Reach for tokens for color/font:

One vendor for all of it. Same SDK, same dashboard. We rejected GrowthBook (PostHog has feature flags + A/B), Hotjar/Clarity (PostHog has session replay + heatmaps), and standalone survey tools (PostHog has surveys).

Install in a client site

cd kamg-ops/sites/<slug>/astro-project

Init in BaseLayout.astro:

---
const POSTHOG_KEY = "phc_xxxx_client_specific_key";
const POSTHOG_HOST = "https://us.i.posthog.com";
---

<script define:vars={{ POSTHOG_KEY, POSTHOG_HOST }}>
  import('posthog-js').then(({ default: posthog }) => {
    posthog.init(POSTHOG_KEY, {
      api_host: POSTHOG_HOST,
      person_profiles: 'identified_only',
      capture_pageview: true,
      capture_pageleave: true,
      session_recording: { maskAllInputs: true }

Lighthouse CI — performance gate in GitHub Actions

Blocks PRs that regress Core Web Vitals. We're running paid traffic to client sites — Quality Score gets crushed if LCP slides past 2.5s.

Workflow file

# .github/workflows/lighthouse.yml
name: Lighthouse CI
on:
  pull_request:
    branches: [main]

jobs:
  lhci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
      - run: npm ci
      - run: npm run build
      - name: Lighthouse CI
        run: |
          npm install -g @lhci/cli
          lhci autorun
        env:
          LHCI_GITHUB_APP_TOKEN: ${{ secrets.LHCI_GITHUB_APP_TOKEN }}

Config — what we enforce

// lighthouserc.cjs
module.exports = {
  ci: {
    collect: { staticDistDir: './dist', numberOfRuns: 3 },
    assert: {
      assertions: {
        'categories:performance':    ['error', { minScore: 0.90 }],
        'categories:accessibility':  ['error', { minScore: 0.95 }],
        'categories:best-practices': ['error', { minScore: 0.95 }],
        'categories:seo':            ['error', { minScore: 0.95 }],
        'largest-contentful-paint':  ['error', { maxNumericValue: 2500 }],
        'cumulative-layout-shift':   ['error', { maxNumericValue: 0.1 }]
      }
    }
  }
};

If LHCI fails the PR — investigate, don't bypass

It's there for a reason. The most common cause is an unoptimized hero image. Use <Image> from astro:assets with explicit width/height + format=avif, and 95% of LHCI failures go away.

Vercel deploy — preview URLs and prod

One-time link, then every push gets a preview URL automatically. Custom domain attaches in dashboard.

Initial wiring (per client repo)

cd kamg-ops/sites/<slug>/astro-project

# 1. log in if you haven't already
vercel login

# 2. link this directory to a Vercel project
vercel link
# Choose the team (kamg / kasandzz), then "Link to existing"
# or create new with the slug as the project name.

# 3. pull env vars (POSTHOG_KEY, etc.) into .env.local
vercel env pull .env.local

# 4. deploy preview manually (optional — every git push does it auto)
vercel

# 5. deploy to prod
vercel --prod

What every PR gets

  • A unique preview URL like redstone-git-feature-quote-form-kamg.vercel.app
  • Automatic Lighthouse run against that URL (via the workflow above)
  • A status check on the PR that links to both

Custom domain

  1. Vercel dashboard → project → Settings → Domains
  2. Add the apex (redstonemfg.com) and www
  3. Add the DNS records Vercel gives you on the registrar (Cloudflare/GoDaddy/whoever the client uses)
  4. Wait for SSL cert (usually 30–90 seconds)

Env vars live in Vercel, not in git

Never commit a .env file. Set vars in Vercel dashboard → Project → Settings → Environment Variables. Pull locally with vercel env pull. We have a separate set per environment (development/preview/production).

The two-remote model

Each client site has two git remotes. Sounds weird, works clean. Solves a real conflict between team visibility and client ownership.

REMOTE 1 — kamg-ops

Tracks the parent sites/<slug>/ dir as part of the shared kamg-ops monorepo. Team can see everyone's client work in one place. Where audit runs and configs live.

cd kamg-ops
git add sites/redstone
git commit -m "redstone: hero copy update"
git push origin master

REMOTE 2 — client

Inside astro-project/ only. Points at the client's own GitHub repo (e.g. kasandzz/redstone-site). This is what Vercel deploys from. Client owns this repo, can be transferred to them later.

cd kamg-ops/sites/redstone/astro-project
git push client main
# → triggers Vercel deploy

Why this shape

  • kamg-ops is for shared work surfaces — sites belong with output/, client-intel/, docs/.
  • Each Astro project still gets its own client GitHub repo — so the client can own their site source, and Vercel has a clean repo to deploy from.
  • Two remotes avoid choosing between team visibility and client ownership.

Push pattern (memorize this)

# after editing files in astro-project/

# 1. push to client repo → triggers Vercel deploy
cd kamg-ops/sites/redstone/astro-project
git add .
git commit -m "feat: services page hero copy"
git push client main

# 2. push to kamg-ops → team can see the change in monorepo
cd ../../../..
git add sites/redstone
git commit -m "redstone: services page hero copy"
git push origin master

If you forget step 2, the team won't see it

Vercel deploys from the client remote, but our monorepo only updates if you also commit at the kamg-ops level. Always do both. I'll add a git hook eventually that does step 2 automatically — for now, manual.

CMS / backend decision tree

Default: no CMS. Astro content collections + Slack request loop covers 80% of clients. The other 20% gets one of two upgrades, never more.

The decision tree

Client situationChoiceReason
Standard service-page site, <10 edits/weekAstro only (file-based)Fastest, $0/mo, no infra
Client wants to self-edit copy without Slack-ing meAdd SanityFree tier covers most cases, hosted, Astro-compatible
Project crosses into app territory: auth, dashboards, member portal, e-commerce, custom workflowsAdd PayloadFull backend, but breaks the locked stack — last resort
Client on WordPress/Webflow and won't migrateHand-build in their platformDon't fight existing CMS
One-off HTML artifact (single email LP, single report)Raw HTML, skip AstroOverkill to scaffold for one file
GHL-hosted pageRaw HTML onlyGHL won't run Astro build output

Why Sanity over Payload as the CMS choice

SANITY (preferred)

  • Free tier covers a typical KAMG site ($0/mo)
  • Hosted — no infra ops
  • Compatible with locked Astro+Vercel stack
  • Real-time collaborative editing
  • Image CDN included

PAYLOAD (last resort)

  • Requires DB + Node server = breaks locked stack
  • Self-host or pay Payload Cloud
  • Right answer ONLY when project = app, not site
  • Stronger when you need: auth, RBAC, admin dashboards, e-commerce

Never Decap CMS

Volunteer-maintained, stability risk for client work. We rejected it.

Backend considerations — when we go beyond static

NeedAdd
Form submissions → emailVercel Form Functions or Formspree
Form submissions → CRMVercel Edge Function → Attio/HubSpot/GHL API
Bookings / calendarEmbed (Calendly/Cal.com) — never build
Auth + portalClerk or Supabase Auth (only if project requires)
DB for dynamic contentSupabase or Vercel Postgres
SearchAlgolia or Pagefind (static-friendly)
Email automationResend (transactional) + client's existing tool

Upsides of staying static + thin-backend

  1. Cost: $0/mo for hosting on most client sites.
  2. Speed: CDN-served HTML beats any rendered framework.
  3. Reliability: No DB, no auth server, no scheduled jobs to break.
  4. Security: No surface area beyond static + a couple of edge functions.
  5. Portability: Client repo is just files — easy to hand off if they leave.
  6. Speed of iteration: Edit Markdown, push, deploy in 30 seconds.

KAMG rules for client work

Non-negotiable on any client-facing build. These are the rules I will reject your PR over.

DOUse design tokens

Always reach for var(--color-*), var(--font-*), var(--space-*). Never hardcode #B21E24 or 16px.

DOComponentize anything that appears 2+ times

Same nav on 12 pages? It's a component. Same CTA on 6 pages? Component. The cost of inlining is high once we need to update copy.

DOMobile-first

90%+ of paid traffic is mobile. Design at 375px first, then scale up. Test every PR on a phone before merging.

DOPreview every PR on mobile

Open the Vercel preview URL on your actual phone, not just dev tools. Catches things devtools misses (touch targets, real font rendering).

DOUse the component library

Before generating any LP/hero/CTA/section: query component-catalog.json or visit /library. Skip this step and you're costing us time.

DOCopy in .md, layout in .astro

Don't mix. Markdown is for content writers, Astro is for structure. They evolve at different speeds.

DON'TUse em dashes in client copy

AI-tell. Use commas, colons, or periods. Yes I know I use them in this doc — internal docs are exempt. Client-facing is not.

DON'TFabricate testimonials or stats

Use only what the client has approved in writing. Inline testimonial quotes are tacky anyway — use pattern-based social proof.

DON'TShip without Lighthouse passing

The CI gate exists for a reason. If LCP is over 2.5s, fix the image — don't merge around it.

DON'TCommit env vars or API keys

Vercel env vars only. Even .env.local — gitignored, never push.

DON'TAdd a vendor without checking with me

Stack is locked. We don't add Hotjar/Clarity/GrowthBook/Mixpanel on top of PostHog. If you think we genuinely need a new vendor, DM me first.

DON'TEdit the starter for client work

The starter is a base. Clients get a copy. Don't push client-specific changes back into the starter.

FTC compliance (any COD-adjacent work)

If you ever do work that touches COD or any client where we make income claims: earnings disclaimers required, no testimonial fabrication, no fake scarcity ("Only 12 spots" ❌, "Limited space" ✅). Income-claim phrasing rules: "sell for $X" / "normalize $X" = ❌; "worth $X" / "build $X program" / "price at $X" = ✅. Different ruleset for Redstone (B2B manufacturing) — looser, but still no fabricated stats.

When to deviate from the stack

There are exactly four legitimate reasons to skip Astro+Vercel+PostHog. Anything else, use the stack.

1. CLIENT IS ON WORDPRESS/WEBFLOW

And won't migrate. Hand-build in their platform. Don't fight an existing CMS that the client is committed to.

2. ONE-OFF HTML ARTIFACT

Single report, single email LP, single thank-you page. Raw HTML, skip the Astro scaffold. Overkill to spin up a project for one file.

3. GHL-HOSTED PAGE

GoHighLevel won't run Astro's build output. Hand-write HTML for anything going into GHL. We do this often for COD niche LPs.

4. APP-TIER PROJECT

Auth, member portal, dashboards, custom workflows. Astro is static-first; if you need a real app, pair with Payload or build separately. Talk to me first.

If you think you have a 5th reason

You probably don't. DM me before deviating. The whole point of locking the stack was to stop re-evaluating per client.
