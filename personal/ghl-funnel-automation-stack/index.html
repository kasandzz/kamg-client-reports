Designed HTML → GHL Page Builder

Turning Claude-designed HTML into native GHL builder pages the client can self-edit.

Practical guide for shipping designed pages into GoHighLevel as proper editable sections, not Custom HTML blobs. What works, why it works, how to set it up, and which adjacent tools to ignore.

How It Works

Import from URL + Playwright cleanup.

GHL's funnel builder ships an "Import from URL" feature that scrapes a public page and rebuilds it as native builder objects: sections, columns, headings, paragraphs, images, buttons, basic styling. The client can click any of them and edit. This is the closest thing GHL exposes to an HTML import API and it's the load-bearing piece of the pipeline.

What Import drops: forms, calendars, popups, JS-driven content, animations, scroll-triggered effects, anything runtime-dependent. Expect ~70% of the page through cleanly, ~30% as GHL-specific gaps.

Playwright MCP (microsoft/playwright-mcp) closes those gaps. It drives an authenticated Chrome session via saved storageState (MFA handled once, persisted), navigates the GHL builder, and applies cleanup deterministically: drop a real GHL Form into the opt-in slot, bind a Calendar widget, set pixels, configure SEO, publish. Claude Code reads the design spec and emits the Playwright calls.

Selector drift is the maintenance cost — GHL ships UI changes ~weekly, expect ~10-20% selector drift per quarter. Pin to data-testid and ARIA roles, plan one refresh sprint per quarter, and accept that the first script costs more than the amortized run.

Terminology

Five terms used throughout this doc.

GHL sub-account
A client's isolated workspace inside GoHighLevel. Each one has its own funnels, forms, calendars, contacts, workflows, and domain mappings. Agencies manage many of these from the agency-level view.
Native sections
Page elements that exist as first-class builder objects: sections containing rows containing columns containing typed elements (Headline, Text, Image, Button, Form, Calendar, Video, etc.). Click-to-edit in the builder UI. The opposite is a Custom HTML element, which renders correctly but isn't section-editable.
Import from URL
A GHL funnel-builder feature. Provide a public URL, GHL fetches the rendered page, and recreates it as native sections inside a new funnel page. The closest thing to an HTML-import API that GHL exposes.
Playwright MCP
An MCP server (microsoft/playwright-mcp) that lets Claude Code drive a real browser via the Playwright API. Useful for any web UI without an automation-friendly API — including GHL's builder. Reuses saved authentication state so it doesn't need to log in every run.
HighLevel Official MCP
GHL's vendor-published MCP server (HTTP, PIT auth). 36 tools covering CRM-side operations: contacts, tags, pipelines, custom fields, calendars, products, media. Does not include funnel/page/form writes — that's the gap Playwright fills.
The Pipeline

End-to-end flow.

1. Claude Code ↓ design HTML/CSS using kamg-astro-starter components ↓ follow import-friendliness rules (see Step 3) 2. Vercel preview deploy ↓ public URL, ~10s 3. GHL → Import from URL ↓ paste preview URL into the funnel builder Import field ↓ GHL renders the page and recreates as native sections 4. Playwright MCP cleanup pass ↓ Claude reads design spec, scripts the GHL-side wiring: ↓ bind Form element, attach Calendar, set pixels, set SEO, publish 5. Client edits in the GHL builder ✓ native section editing works as expected
Setup

Order of operations.

1

Install the two MCP servers

Add to ~/.claude.json: microsoft/playwright-mcp (browser automation) and the HighLevel Official MCP (HTTP, vendor-published). Generate a Private Integration Token in the client's GHL sub-account (Settings → Private Integrations), scope it narrowly, store in .env.

2

One-time GHL login → persist storageState

Open a headed Playwright session, log into the GHL sub-account by hand (with MFA), save cookies + localStorage to auth-state.json via Playwright's browser_storage_state. Gitignore the file. Every subsequent run loads this state and runs authenticated without prompts.

3

Design the page with import-friendliness in mind

Import-from-URL works on the rendered DOM. Anything that depends on JS execution, scroll position, hover, or runtime fetches gets dropped or flattened. Discipline:

  • Real semantic <section> structure, one logical block per section
  • Standard fonts via Google Fonts; avoid exotic webfonts or font-variable tricks
  • Static content only — no JS-rendered headings, no client-side data fetches
  • CSS grid and flexbox fine; absolute positioning and CSS transforms confuse the import
  • No scroll-triggered animations, parallax, intersection-observer effects
4

Deploy to a public URL

Vercel preview, Netlify, GitHub Pages — any public host. GHL needs to fetch the rendered page over the open internet, so localhost won't work. Vercel previews tear down after the build so leakage isn't a concern for short-lived imports.

5

Import inside GHL

In the client sub-account: Funnels & Websites → New Funnel → Add Step → Import from URL. Paste the preview URL, wait ~30-60s for the scrape and conversion. Expect to land with: layout, headlines, body copy, images, buttons as native elements. Expect to be missing: working forms, calendars, checkout, popups, anything dynamic.

6

Run the Playwright cleanup script

Claude Code reads the original design spec (which forms go where, which calendar, which workflow on submit, tracking pixels, SEO meta) and drives the GHL builder via Playwright MCP to apply the GHL-specific wiring. Per-section it: drops the right native element, opens its config panel, selects the bound resource, saves.

Build the script once per client vertical or page archetype. Subsequent funnels in the same shape become near-instant runs.

7

Hand off

Client logs in, clicks any headline, edits in place. Each section is a real builder section so the experience matches what they signed up for when they bought GHL.

Tradeoffs

What this approach costs.

CostWhyMitigation
Design fidelity drift Import flattens CSS transforms, custom font stacks, complex grids. Result is "close to" the source, not pixel-perfect. Design import-friendly from the start. Treat the import as the baseline, polish via the builder or custom CSS panel.
First-build cost is high Playwright cleanup scripts have to be written and stabilized before they save time. Write the script during the first funnel of a given vertical. Amortize across subsequent builds.
Selector drift GHL ships UI changes weekly. Expect ~10-20% of builder selectors to drift per quarter. Pin to data-testid and ARIA roles. Run a selector refresh sprint each quarter. Avoid XPath and text-content navigation.
Irreducibly manual integrations Stripe / Mailgun / Twilio / GA / Meta connections all require interactive OAuth consent per sub-account. DNS for custom domains lives at the registrar. Budget one "integrations day" per new client. No automation path eliminates these.
Adjacent Tools, And Why They're Not Here

What I cut from earlier versions of this doc.

Earlier drafts included these. Each has legitimate use cases — none of them serves the goal of "designed HTML → editable GHL page."

Snapshots
What it is: A GHL feature that clones an entire sub-account configuration (funnels, workflows, forms, custom fields, templates) into another sub-account. Why it's not here: Snapshots replicate things you've already built inside GHL. They don't ingest HTML. Different problem.
n8n + n8n-mcp
What it is: Workflow automation platform for connecting apps via webhooks and event chains. Why it's not here: n8n is post-deployment glue — form-submit triggers, CRM sync, notification fan-out. It doesn't translate page markup into a builder.
External Astro hosting
What it is: Host the LP on Vercel, route GHL forms via webhooks. Why it's not here: The page no longer lives in GHL, so client self-edit in the GHL builder is impossible. Fails the core constraint.
Funnel AI as translator
What it is: GHL's built-in AI accepts a URL or screenshot as visual inspiration and regenerates a page. Why it's not here: It's a regenerator, not a converter. Your design becomes mood-board input. Result is generic, not your design. Different tool, different job.
Claude Chrome extension
What it is: Anthropic's browser-using Claude (Pro / Max), drives an authenticated tab. Why it's not here: Useful for GHL admin work — smart lists, tag cleanup, workflows config — but documented to struggle with drag-drop canvas builders. Wrong tool for the page-builder write layer.
Visual-diff fix loop
What it is: Render reference + actual, pixel-diff, send to Claude, emit fix script, iterate until convergence. Why it's not here: Overengineered for current volumes. Import + Playwright cleanup gets close enough. Revisit if shipping 30+ identical pages where pixel parity is contractual.
Bottom Line

The answer is more boring than the question implied.

"Build entire GHL funnels with Claude Code + MCPs" sounds like there should be a generative API or a hot new agent that one-shots the whole thing. There isn't. GHL deliberately doesn't expose the page builder, and no MCP changes that.

What's actually available — Import from URL + Playwright MCP + design discipline — is enough to compress a 6-8 hour manual build to under an hour, repeatable across clients, with the client retaining full self-edit in the GHL builder. The first build is the expensive one; everything after that runs against the same script.

The week-one move: ship one funnel end-to-end through this pipeline. Pick a real client. Design the HTML with the import-friendliness rules. Deploy. Import. Write the cleanup script as you go. Hand off. Note what broke, fix the script, ship the second build at half the cost.
