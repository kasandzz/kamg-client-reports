Building GHL funnels with Claude Code: what's real, what's myth, what's worth building.

An honest look at five candidate paths for automating funnel production inside GoHighLevel via Claude Code, MCPs, and browser automation. No single path wins on every axis. The point of this brief is to map the tradeoffs so the right path can be matched to each build.

RESEARCH · 6 parallel agents SOURCES · 40+ vendor & community refs SCOPE · Practical agency application
The Premise

Funnel production is the agency bottleneck, but the obvious automation paths each have non-obvious limits.

A typical client funnel inside GHL involves roughly 45-55 discrete configuration steps across pages, sections, forms, surveys, calendars, workflows, triggers, tags, pipelines, templates, domains, pixels, and payment connections. Manual build time runs 6-8 hours per funnel. With multiple active clients shipping funnels regularly, the math gets uncomfortable fast.

The instinct is to reach for the API or the latest agent. The honest finding is that GHL's API deliberately doesn't expose the page builder, Funnel AI's HTML-paste workflow is largely a community myth, and Claude Chrome chokes on drag-drop canvases. None of which means automation is dead. It means the architecture has to compose multiple imperfect tools.

"What's the most practical way to apply Claude Code + MCPs + browser agents to GHL funnel production, given that each tool has real limits?"

Myth-Bust First

The "paste HTML into GHL Funnel AI" workflow.

Common claim online: paste HTML into GHL's Funnel AI and get a 70%-built funnel. The research found this is a conflation of three separate features.

What's Actually True

Funnel AI ingests URL or screenshot, for style inspiration only. It does not convert HTML markup into native sections.

Per GHL's help documentation and Labs changelog, Funnel AI runs in four modes (Assist, Build, Chat, Import-from-Link/Image). The Import mode takes a URL or screenshot and uses it as visual inspiration, then regenerates page content from scratch. It does not parse HTML, CSS, or component structure from the source.

The "70% built from HTML" claim circulating online appears to conflate three separate features:

  • Import from URL · scrapes a published page. Sections/columns/images survive. Forms, calendars, scripts, popups all break and rebuild manually.
  • Funnel AI's URL/image input · visual inspiration only. The AI regenerates from scratch; your reference page becomes a vibes-input, not a spec.
  • Custom HTML/JS element · paste markup as a single opaque "blob." Renders fine, but the region is not editable in the visual builder and can't bind GHL forms/CRM events inside without bridging via custom event handlers.

The native HTML-import feature request HIGHLVL-I-2696 has sat open on the ideas board since January 2021 with no official response. Realistic Funnel AI output is "draft scaffold worth iterating," not "70% finished build."

The Five Candidate Paths

Each one is a real workflow. None of them is universally correct.

Ordered roughly from most mature to most speculative. Pros, cons, and best-fit conditions for each.

PATH 01

Snapshot Clone + CustomizeThe boring path that actually wins most of the time

Build a master snapshot once in the GHL UI per client vertical (education, fitness, professional services, etc.). Push to each new sub-account. Then layer in customization via API and MCP for tags, pipelines, custom fields, contacts, calendars, and media.

Strengths

  • ~70% of funnel structure carries cleanly
  • Native editability preserved end-to-end
  • Maintainability survives weekly GHL UI changes
  • One-time IP investment scales infinitely

Limits

  • Excludes Stripe/Mailgun OAuth, domains, phone numbers
  • Snapshot creation is UI-only (no API)
  • Quality is bounded by the master template's quality
  • Vertical-by-vertical investment upfront

Bonus Path · Speculative

The visual-diff fix loop: interesting, probably overengineered.

Could Playwright compare the GHL-rendered output against a reference HTML and surface deltas for Claude to fix? Research finding: technically buildable, but the closed-loop version is overengineered for 5-10 funnels/month. There's an interesting middle path worth keeping.

The Middle Path That Actually Earns Its Complexity

Snapshot both pages → diff with odiff/pixelmatch → send to Claude for plain-English delta report → stop there.

Don't build the closed loop where Claude autonomously emits Playwright fix scripts and iterates. That's where the cost blows up: ~20-40 min wall-clock per page, $0.50-1.50 in API cost per iteration, and you spend more time debugging false positives from font-loading races and CMS wrappers than shipping pages.

Build steps 1-4: take the screenshots, run the diff, ask Claude "list the visual deltas in plain English," and hand the report to a human (you, or future-you) to apply manually in the GHL builder. 80% of the value at 20% of the engineering cost.

The closed loop earns its complexity only when (a) you'll run it >30 times before it's obsolete, and (b) the human polish step it replaces is >30 min per page. Below that threshold, a Playwright snapshot + manual eyeball + targeted CSS fixes in the builder's custom-CSS panel beats it on every axis.

The Three-Way Tradeoff

Pick two of three.

This is the honest constraint underneath every path.

There is no current path that delivers all three of: Astro-grade design fidelity, native GHL editability, and full CRM/form/checkout binding, in one workflow.

Design Fidelity

Premium brand polish, performance, component-grade design system

Native Editability

Client can self-edit in GHL UI, sections/rows/elements are first-class

CRM + Form Binding

GHL forms, calendars, checkouts, workflows fully wired in-page

Snapshot + MCP + Playwright gets you native editability + CRM binding. Pays in bounded design ceiling.

External Astro + webhook glue gets you design fidelity + CRM binding (via webhooks). Pays in losing native editability.

Funnel AI scaffold + manual polish gets you native editability + CRM binding. Pays in generic-feeling design ceiling and human polish time.

The strategic question isn't "which is best," it's "which two does this specific build need most." A premium one-off for a high-paying client probably weights design + CRM. A mass-produced funnel for a templated vertical probably weights editability + CRM. Choose deliberately.

Practical Considerations

Things to weight before committing to a stack.

Surface-level "which tool" comparison misses the operational reality.

UI Drift

GHL ships UI changes weekly. Every browser-driver path inherits this maintenance tax.

Playwright MCP selectors and Claude Chrome scripts will drift roughly 10-20% per quarter. Budget for selector refresh sprints. Pin to data-testid where possible, avoid XPath, never depend on text content for navigation.

Client Self-Edit

If the client expects to log in and edit text, native paths win automatically.

External Astro pages are git-deployed. Every client text change becomes a developer task. Worth confirming with each client whether self-edit matters before recommending Path 5.

OAuth Reality

Stripe, Mailgun, Twilio, GA4, Meta connections require human consent screens. Per sub-account.

No automation path eliminates this. Plan a "manual integrations day" per new client regardless of which technical path the rest of the build takes.

Brand Voice + Copy

Funnel AI's draft copy is generic. Snapshot copy is templated. External Astro copy is whatever you wrote.

None of the paths solve "good copy." Plan a copy pass at the end of any path. Consider building a copy-brief skill that feeds an LLM with brand voice + product context before any technical build kicks off.

Auth + Permissions

Private Integration Tokens (PIT) are scoped per location, max 5 per location.

Generate PITs at the narrowest viable scope per use case. Rotate every 90 days. Never check PITs into git. Use .env + gcloud secret manager pattern already in place for COD.

Maintenance Cost

Build cost is one-time. Maintenance cost compounds.

Snapshot maintenance: low (UI refresh once per quarter). MCP maintenance: low (vendor-maintained). Playwright maintenance: medium (selector drift). Closed-loop visual-diff: high (false positives, model drift). Match the maintenance budget to the build volume.

Where to land

The honest position isn't "build this stack." It's "stop chasing the API and start composing the paths."

The strongest practical move is to build out Path 1 (snapshot library) + Path 2 (Official MCP) first, because they cover the largest share of any build with the lowest maintenance cost. Add Path 3 (Playwright MCP) when a specific workflow demands UI-only writes you can't dodge. Reach for Path 4 (Funnel AI) only when starting from zero in a new niche. Reserve Path 5 (Astro external) for premium one-offs where design fidelity outweighs native editability.

The myth-bust is the most valuable finding: do not expect to paste HTML into GHL Funnel AI and get a finished funnel. That workflow does not exist as advertised. The closest realistic version is "deploy an Astro page publicly first, feed its URL to Funnel AI as visual inspiration, accept the regenerated draft, polish manually," which is several steps short of "convert this HTML."

The visual-diff fix loop is intellectually interesting and worth keeping in the back pocket. Build the open version (delta report, no autonomous fix) before considering the closed version. Most agency volumes don't justify the closed loop.

The largest unresolved question isn't technical, it's strategic: how much of the agency's value lives in the funnel-build motion versus the strategy/copy/offer work that surrounds it? If automation collapses funnel build time from 6 hours to 1 hour, that hour either goes to higher-leverage work or to lower pricing. Worth deciding in advance which.
