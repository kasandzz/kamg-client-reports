Designed HTML → GHL Page Builder

How to take a page you designed in Claude Code and get it inside GHL's funnel builder so the client can edit it themselves.

Plain-English guide. What you want, why it's hard, what actually works, how to set it up. Cuts everything that doesn't serve this specific goal.

The Goal in Plain English

You want this. Not anything else.

You design a beautiful landing page in Claude Code. Real HTML, real CSS, maybe pulling from your kamg-astro-starter components. It looks great in the browser.

Now you want that page living inside your client's GoHighLevel account, sitting in their funnel builder as proper editable sections. So when the client logs in later, they can change the headline, swap an image, or update copy themselves, without calling you.

That last part is the whole point. If the result isn't editable in the GHL builder, you've failed the brief. The client wants control of their own page.

First, the basics

What GHL actually is, in human language.

GoHighLevel (GHL) is a marketing platform agencies resell to clients. Think Squarespace + Mailchimp + Calendly + Stripe + a CRM, all in one dashboard. Each client gets a "sub-account," their own little workspace.

Inside that workspace is the Funnel & Website Builder. It's a drag-and-drop page editor, exactly like Squarespace, Wix, or Webflow. Pages are built from "sections," which contain "rows," which contain "columns," which hold "elements" like text, images, buttons, forms.

When something is built as native sections, the client can click on any element in the builder, type new text, swap an image, drag things around. That's the editability we're after.

The Trap to Avoid

If you paste your HTML into a "Custom HTML" element, the page renders fine, but it becomes one big un-editable blob.

The client sees the page but can't click on the headline to change it. They'd have to know how to edit raw HTML. That defeats the entire reason they're using GHL. So Custom HTML paste is off the table for our goal.

The plot twist

There's a button hiding in GHL that does most of the work.

GHL has a feature called "Import from URL." You point it at a published web page, and it scrapes that page into native GHL sections automatically. Sections, columns, images, basic styling, all imported as proper editable builder elements.

This is the load-bearing fact for our entire plan. The official GHL documentation confirms it: imports sections/columns/images/basic styling as native elements. Some things break, forms, calendars, popups, complex animations, and those get rebuilt by hand. But the layout and copy come through.

This is the bridge from your Claude-designed HTML to the GHL builder. You design in Claude Code, host the page somewhere public (Vercel preview is fine), then point GHL's "Import from URL" at it. Native sections come out the other end.

What This Means For You

Your real job becomes: design HTML that imports cleanly.

Simple semantic HTML with clean CSS will import well. JavaScript-driven content, complex animations, and CSS magic that depends on hover/scroll triggers will not. Design for import-friendliness up front and the rest is much smoother.

Where Playwright comes in

The robot that fixes what Import-from-URL misses.

Import-from-URL gets you to ~70% of a finished page. The remaining ~30% is GHL-specific stuff that doesn't exist in your reference HTML: form bindings, calendar widgets, checkout elements, conversion pixels, the publish button.

That gap is where Playwright MCP comes in. Plain English: Playwright is a piece of software that drives a real web browser like a human would, clicking buttons, typing into fields, dragging elements. MCP ("Model Context Protocol") is the bridge that lets Claude Code tell Playwright what to do.

So Claude reads your design spec, opens the GHL builder via Playwright, and clicks through the cleanup steps: "drag a form element here, connect it to the lead-capture workflow, set the redirect URL, click publish." Same thing you'd do by hand, except scripted so it runs in 2 minutes instead of 20.

Glossary · ELI5

The five terms you actually need to know.

GHL sub-account
Your client's workspace inside GoHighLevel. Each sub-account has its own funnels, forms, calendars, contacts. You manage these as the agency from your master "agency view."
Native sections
Page elements (headline, image, button, form) that exist as first-class objects inside the GHL builder. Click-to-edit. Opposite of a Custom HTML blob.
Import from URL
A GHL builder feature. Paste a public web URL, GHL scrapes the page and recreates it as native sections inside a new funnel page. Built-in HTML-to-builder converter.
Playwright
Software that drives a real browser like a remote-controlled mouse and keyboard. Click here, type that, upload this file. We use it to script the cleanup work in the GHL builder after Import-from-URL.
MCP server
A small piece of software that lets Claude Code use external tools. Playwright MCP = Claude can drive Playwright. HighLevel Official MCP = Claude can read/write the GHL API (contacts, tags, pipelines).

Step-by-Step Setup · ELI5

What to actually do, in order.

1

Get the basics installed in Claude Code

Two MCP servers, one-time setup. Add both to your ~/.claude.json file.

  • Playwright MCP (microsoft/playwright-mcp) — the browser robot
  • HighLevel Official MCP (HTTP, vendor-provided) — for any API work (contacts, tags, pipelines)

Plain English: these are the tools Claude needs to talk to GHL and to drive a browser.

2

Log into GHL once, save the session

Open a real Chrome window via Playwright. Log into the client's GHL sub-account by hand once (with MFA if needed). Save the browser session to a file (auth-state.json). Playwright reuses this every future run, so you never have to log in again.

Plain English: like saving your Netflix login on a TV. You do it once.

3

Design the page in Claude Code with import in mind

Build the page as you would normally, but with discipline:

  • Use clear semantic HTML (real <section> tags, clear hierarchy)
  • Stick to standard fonts (Google Fonts is fine, no exotic webfonts)
  • No JavaScript-driven content, everything must render in static HTML
  • Avoid complex animations or scroll-triggered effects
  • Use simple CSS, grid and flexbox are fine, absolute positioning will confuse the import

Plain English: design like it's 2014. Boring HTML imports cleanly. Fancy HTML breaks.

4

Deploy to a public URL

Push to a Vercel preview, Netlify drop, or GitHub Pages. Anywhere public. Takes 30 seconds. You just need a URL GHL can fetch.

Plain English: GHL needs to be able to visit your page on the open internet to copy it.

5

In the client's GHL: Import from URL

Funnels & Websites → New Funnel → Add Step → choose "Import from a URL" (or Import button in the page builder). Paste your Vercel URL. Wait ~30-60 seconds. GHL scrapes the page into native sections.

You'll get headlines, paragraphs, images, buttons, and basic layout as real builder elements. You'll be missing: working forms, calendars, checkout, popups, anything dynamic.

Plain English: this is the magic step. The button you didn't know existed.

6

Playwright cleanup pass

Ask Claude Code to run the cleanup script. It opens the GHL builder via Playwright (using your saved login) and methodically:

  • Drags a GHL Form element into the opt-in slot, picks the form from the dropdown
  • Drops a Calendar element into the booking slot, picks the calendar
  • Adds tracking pixels in head/body via the page settings panel
  • Saves and publishes

Plain English: the robot does the dragging-and-clicking work that you'd do by hand. Same result, 10x faster.

7

Hand off to the client

Client logs in. They can click any headline and retype it. They can swap any image. They can rearrange sections. Goal achieved.

Plain English: you've delivered a beautifully-designed page that's also fully editable in their own dashboard.

Honest Tradeoffs

What this approach actually costs you.

What you give upWhy it happensHow to mitigate
Pixel-perfect design fidelity Import-from-URL flattens fancy CSS. Custom fonts may swap. Complex layouts may simplify. Design import-friendly from the start. Treat the result as a "near-match" not "exact match."
Speed on the first run Playwright scripts need to be written and tested before they save time. Build the cleanup script once per client vertical. Reuse it across funnels.
Maintenance discipline GHL ships UI changes weekly. Your Playwright selectors may drift 10-20% per quarter. Pin to stable selectors (data-testid, role-based). Plan one selector-refresh sprint each quarter.
Some things still must be manual OAuth connections (Stripe, Mailgun, Twilio), DNS for custom domains, payment-method selection. Accept "one manual integrations day per new client." No automation eliminates these.
What got cut from v1/v2 of this brief

Tools and paths that sound related but don't serve this goal.

Including them confused the picture. Here's what each one actually does, and why it doesn't fit "designed HTML → editable GHL page."

Snapshots
What it does: Clones an existing GHL workspace (funnels + workflows + forms + everything) to a new sub-account. Why it doesn't fit: Snapshots clone GHL builds you already made inside GHL. They don't translate designed HTML into GHL. Wrong direction entirely — useful for replicating a working setup across multiple clients, not for getting a new HTML design into GHL.
n8n + n8n-mcp
What it does: Automation platform for connecting apps together (form-submit → Slack message → CRM update). Why it doesn't fit: n8n is about gluing systems together for ongoing workflows. It has zero to do with translating HTML into a page builder. Useful for other agency work, irrelevant here.
External Astro hosting
What it does: Host the LP on Vercel and connect GHL forms/calendars via webhooks. Why it doesn't fit: The page lives on Vercel, not in GHL. Client can't self-edit in the GHL builder anymore — every text change becomes a developer task. Fails the core goal.
Funnel AI as a translator
What it does: GHL's built-in AI takes a URL or screenshot as visual inspiration and regenerates a page from scratch. Why it doesn't fit: It's not a translator, it's a regenerator. Your carefully-designed HTML becomes a vibes-input. Result is generic, not your design. Useful for net-new niches with no reference, not for shipping your specific design.
Claude Chrome extension
What it does: Anthropic's browser-using Claude, drives your authenticated tab. Why it doesn't fit (yet): DOM-driven, chokes on drag-drop canvases like GHL's builder. Documented case of a 10-minute task taking 2 hours. Useful for GHL admin work (smart lists, tags, workflows config). Not for the builder canvas.
Visual-diff fix loop
What it does: Renders both pages, takes screenshots, asks Claude to spot differences, scripts fixes. Why it doesn't fit: Overengineered for the volume. The Import-from-URL → Playwright-cleanup pipeline gets you to "close enough" without the closed loop. Reach for this only if you're shipping >30 of the same page and pixel parity is contractual.
The Honest Bottom Line

If this approach feels lower-tech than expected, that's because it is.

The "build entire funnels with Claude Code + MCPs" framing makes it sound like there's a magic API or a hot new agent that does the whole thing in one prompt. There isn't. GHL deliberately doesn't open its page builder to outside automation, and no MCP can change that.

What's actually available is more boring and more useful: a built-in import feature most agencies don't know about, a browser robot that closes the gaps, and discipline in how you design the source HTML. That stack is enough to take your designed page into an editable client builder in well under an hour, repeatedly.

The week-one move is to ship one real funnel through this pipeline end-to-end. Pick a real client. Design the page in Claude Code with import-friendliness rules. Deploy to Vercel. Import-from-URL in GHL. Write the Playwright cleanup script. Hand off to the client. Then watch what actually breaks, fix it, save the script, and the second funnel is half the work of the first.
