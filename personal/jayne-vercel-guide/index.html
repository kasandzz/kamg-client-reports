Why the browser flow, not the CLI

The CLI deploys, but it doesn't auto-wire push-to-deploy

Counterintuitive but true: vercel link + vercel --prod creates a project that ships from your laptop, not one that auto-rebuilds on every GitHub push. Getting the "edit code, say commit and push, site updates 60 seconds later" workflow requires the project be created via the Import Repository button in Vercel's web dashboard. That's the next step.

If your registrar supports ALIAS / ANAME records (Cloudflare's "CNAME flattening", Cloudflare Registrar, DNSimple, Namecheap, Porkbun all do), prefer that over the bare A record for root domains. ALIAS records follow Vercel's internal IP rotations automatically; the A record locks you to whatever IP is current today.

5.Add DNS records at your registrar

The part that varies by where you bought your domain. Use the exact values from Vercel's Domains page, not the placeholders below. Three common registrars, plus a fallback Claude prompt for everything else.

If your domain is on Cloudflare

  1. Log in at cloudflare.com, click the domain.
  2. Left sidebar: DNS → Records → Add record.
  3. Match what Vercel showed you exactly: Type (CNAME or A), Name, and Target/Content.
  4. Toggle the orange cloud to grey ("DNS only"). Single most common Cloudflare trip. Orange cloud terminates TLS at Cloudflare's edge and blocks Vercel from issuing its own SSL cert. Full/Full-strict modes do not fix this; the cloud has to be grey.
  5. Save.

For root domains on Cloudflare: use CNAME with Cloudflare's automatic CNAME flattening (just add a CNAME at @, Cloudflare handles the apex rewrite).

If your domain is on Namecheap

  1. Domain List → Manage → Advanced DNS tab.
  2. Add New Record. Type: CNAME Record (or A, per Vercel). Host: your subdomain (e.g. dashboard) or @ for root. Value: whatever Vercel showed you. TTL: Automatic.
  3. For root domains, Namecheap supports ALIAS Record. Prefer ALIAS over A if Vercel offers a CNAME-style target for the root.
  4. Green checkmark to save.

If your domain is on Porkbun

  1. Domain Management → click the domain → DNS Records.
  2. Add. Type: CNAME or ALIAS (Porkbun supports both). Host: subdomain or blank for root. Answer: whatever Vercel showed you.
  3. Save.

If your domain is somewhere else

Five to thirty minutes is common. Worst case 24-48 hours if your TTL is high or your registrar's caches are slow. Use dnschecker.org to watch propagation. Back in Vercel's Domains page, the status moves Invalid Configuration → Pending → Valid Configuration ✓. SSL is automatic. Vercel issues a Let's Encrypt cert the moment DNS resolves.

The Cloudflare trip

Grey cloud, not orange

If your domain's DNS is on Cloudflare and you leave the proxy cloud orange, your domain appears to work but Vercel can't issue its SSL cert and visitors get "Not Secure" warnings. Grey cloud = DNS only = works with Vercel SSL. Cloudflare's "Full / Full-strict" SSL modes do not fix this; the proxy itself has to be off.

The env-var race

Add the vars in Vercel BEFORE the code reaches it. Then redeploy.

Vercel env vars only apply to new deployments. If the auth code pushes and starts deploying at moment T, but you add the env vars at T+10s, that deployment may have already snapshotted env and won't have your credentials, locking everyone out including you. Two protections: add the env vars first, then push. After push, manually Redeploy from Vercel's Deployments tab to be 100% sure the new env values are picked up.

How to add the env vars in Vercel

  1. Vercel project → Settings → Environment Variables.
  2. Click in the Key field, type DASHBOARD_USER.
  3. Click in the Value field, paste the username Claude gave you.
  4. Make sure Production, Preview, Development are all checked.
  5. Click Save.
  6. Repeat for DASHBOARD_PASS.

Optional · After it's working

Production polish

Stuff worth doing once the dashboard is live, password-protected, and on your domain. Each item below is independent. Skip anything you don't need. The big paste-prompt at the end batches the common ones.

Cache headers (fast loads, low Vercel function usage)

Vercel's defaults are fine for most sites, but if your dashboard pulls live data on every load (Q4=a in the calibrate step), you'll burn through Hobby's 100,000-function-invocation limit quickly without caching. Add a vercel.json with cache rules: long-cache for static assets (fonts, images, JS bundles, anything fingerprinted), short-cache for HTML, and explicit s-maxage + stale-while-revalidate for API responses that don't need to be live-live.

Lazy-load heavy media

Two near-free wins for dashboards with charts, screenshots, or hero imagery: add loading="lazy" to every <img> below the fold, and code-split heavy components (chart libs, PDF viewers, anything >100KB) so the initial page paints fast. For Next.js: next/dynamic. For Astro: client:visible. For Vite/React: React.lazy + Suspense.

Mobile test (90% of dashboard traffic on phones)

If this dashboard is going to anyone other than you, assume the majority will open it on a phone first. Pull it up on your actual phone. Then open Chrome DevTools, toggle device mode, click through every interaction. Fix the obvious tap-target and text-size issues before sharing.

Lighthouse score

In Chrome DevTools → Lighthouse tab → Generate report. Aim for Performance > 90 on mobile. Anything red, paste the audit to Claude with "fix these". Lighthouse will catch unused JS, oversized images, missing alt text, render-blocking resources, the common stuff Claude can fix without you having to know what they mean.

Analytics (optional, depending on Q5)

If you said yes to analytics in the calibrate step: Plausible for traffic-only (privacy-friendly, GDPR-clean, free up to 10K events/mo), PostHog for behavior tracking with session replays (free up to 1M events/mo). Both are one script tag in the page head, Claude wires either in five minutes.

The other Kas-ism worth knowing

Vercel keeps every deployment, use it

If anything in the polish pass makes the site feel slower or breaks something subtle, Vercel's Deployments tab is your time machine. Click the three-dot menu on any prior deployment and "Promote to Production". Instant rollback, no code change required. This is also your safety net for any future risky push.

When something breaks

The four things that go wrong most often, and what to do.

"My deployment failed in Vercel"

Click into the failed deployment in Vercel's Deployments tab. Scroll through the build log until you see red text. Copy the red text. Paste it to Claude with "Vercel build failed with this, fix it". Claude reads the error and patches the code. Then push again.

"My domain isn't loading / says 'not secure'"

Wait. DNS is slow. Check dnschecker.org with your domain. If it's not green everywhere yet, give it another hour. If after a few hours it's still broken: in Vercel, your domain page will show what's wrong (usually a wrong DNS record). Copy the exact error message into Claude and paste your DNS records (with secrets removed). Claude will tell you what to fix.

"The password prompt isn't appearing"

Two usual reasons: (1) you forgot to add the env vars in Vercel, or (2) you added them but Vercel hasn't redeployed since. Fix: in Vercel → Deployments tab → click the three-dot menu on the latest deployment → Redeploy. Or push any small change.

"I can't get past my own password prompt"

You typed the wrong username/password. Vercel → Settings → Environment Variables shows you what DASHBOARD_USER is set to (click the eyeball to reveal). For the password, you'll need to either remember it or replace it. To replace: edit the DASHBOARD_PASS value, save, then in Deployments tab → redeploy. New password active in 60 seconds.

When in doubt

Paste the symptom to Claude

You don't have to diagnose anything yourself. "The site shows X, what do I do" is enough. Claude can read the build logs, the config, and the error, it's a much better debugger than I am at 3am.

The host. The company whose computers actually serve your site to anyone who visits the URL. Free tier covers everything in this guide.
Deploy
"Make the changes live." Vercel takes your latest code from GitHub, builds it, and puts the result on the internet at your URL.
DNS
The internet's address book. Tells browsers where to send requests for your domain. Lives at your domain registrar (Cloudflare, Namecheap, etc).
Registrar
The company you bought your domain from. Manages DNS for that domain.
CNAME / A record
Types of DNS records. A CNAME points one domain at another domain. An A record points at a specific IP address. Vercel tells you which one to add.
SSL / HTTPS
The padlock in the address bar. Vercel sets this up automatically and for free. You don't have to think about it.
Env var
Environment variable. A secret value (like a password) that your code reads at runtime. Stored in Vercel, never in your code, never on GitHub.
Push
Upload your latest code to GitHub. The git command. Triggers Vercel to redeploy.
Commit
A snapshot of your code. One unit of change with a message. You commit before you push.
