Before You Start

Three accounts, fifteen minutes once

Open these in three tabs and get the signups out of the way. Everything in this guide assumes they exist.

A.

GitHub

If you don't already have one, sign up at github.com. Free plan. Private repos are unlimited, the dashboard never goes public.

B.

Vercel

Sign up at vercel.com. On the signup screen, choose Continue with GitHub, not email, not Google. This is what lets Claude (via the Vercel CLI) link your account without an extra OAuth dance. Pick the free Hobby plan when prompted.

C.

A domain

Either one you already own (you'll use a subdomain like dashboard.yoursite.com) or a fresh one from Namecheap, Cloudflare Registrar, or Porkbun. About $12/year. Avoid GoDaddy, its DNS panel is unnecessarily painful.

1.Open Claude Code in the dashboard folder

Same way you've been opening it for everything else, terminal in the project root, run claude. The two prompts below assume the working directory is your dashboard folder. If it isn't, cd there first.

2.Push to GitHub + deploy to Vercel

This is the big one. One paste, and Claude takes care of: detecting the framework, initializing git, creating a private GitHub repo, pushing the code, installing the Vercel CLI, logging into Vercel, linking the project, and deploying. End state: a working *.vercel.app URL. Two browser pop-ups in the middle (one for GitHub auth, one for Vercel auth). Claude pauses while you handle them.

Two browser pop-ups

One for GitHub, one for Vercel

Claude will pause twice. First when gh auth login opens GitHub (you paste an 8-char code and click Authorize). Second when vercel login opens Vercel (you click "Continue with GitHub" and confirm). Both take ~10 seconds. Claude waits.

3.Add your custom domain in Vercel

Two minutes of clicking, then Vercel will give you DNS records to copy. Keep this tab open, you'll paste those records into your registrar in Step 4.

  1. Open vercel.com, click into the project you just deployed.
  2. Top nav: Settings → left sidebar: Domains.
  3. Type the domain you want (e.g. dashboard.yoursite.com for a subdomain, or yoursite.com for the root). Click Add.

4.Add DNS records at your registrar

The part that varies by where you bought your domain. Three of the most common, plus a fallback Claude prompt for everything else.

If your domain is on Cloudflare

  1. Log in at cloudflare.com, click the domain.
  2. Left sidebar: DNS → Records → Add record.
  3. Match what Vercel showed you (Type, Name, Target).
  4. Toggle the orange cloud to grey ("DNS only"). This is the single most common Cloudflare trip, orange cloud breaks Vercel's SSL issuance.
  5. Save.

If your domain is on Namecheap

  1. Domain List → Manage → Advanced DNS tab.
  2. Add New Record. Type: CNAME Record. Host: your subdomain (e.g. dashboard). Value: cname.vercel-dns.com. TTL: Automatic.
  3. Green checkmark to save.

If your domain is on Porkbun

  1. Domain Management → click the domain → DNS Records.
  2. Add. Type: CNAME. Host: subdomain. Answer: cname.vercel-dns.com.
  3. Save.

If your domain is somewhere else

Hand this off to Claude, it's a five-second lookup, not a guide-worthy section:

Then wait for DNS

Five to thirty minutes is typical. Use dnschecker.org to watch the propagation. Back in Vercel's Domains page, the status will go Invalid Configuration → Pending → Valid Configuration ✓. SSL is automatic. Vercel issues a cert as soon as DNS resolves.

The Cloudflare trip

Grey cloud, not orange

If your domain's DNS is on Cloudflare and you leave the proxy cloud orange, your domain appears to work but Vercel can't issue an SSL cert and visitors get "Not Secure" warnings. Grey cloud = DNS only = works.

The order matters

Add env vars BEFORE pushing the code

If you push the code before adding DASHBOARD_USER and DASHBOARD_PASS in Vercel, your site will lock everyone out (including you) for the 30 seconds it takes to add them. Not a disaster, but annoying. The prompt above forces Claude to wait, humor it.

How to add the env vars in Vercel

  1. Vercel project → Settings → Environment Variables.
  2. Click in the Key field, type DASHBOARD_USER.
  3. Click in the Value field, paste the username Claude gave you.
  4. Make sure Production, Preview, Development are all checked.
  5. Click Save.
  6. Do it again for DASHBOARD_PASS.

Your new workflow, forever

Setup was a one-time thing. From now on, the loop is short:

1.

You edit the dashboard with Claude Code as normal

Make changes, ask Claude to fix things, add features. Same as you've been doing.

2.

When you're happy, tell Claude: "commit and push"

Claude runs the three git commands (add, commit, push) and your code goes to GitHub.

3.

Vercel auto-detects and rebuilds

Within seconds of the push, Vercel sees it and starts building. 30–90 seconds later, your live site is updated. You don't have to do anything, no upload, no FTP, no "deploy" button.

4.

Refresh your domain to see the changes

That's it. Three words, sixty seconds.

Bonus

Every push is also a backup and a safety net

Because all your changes are in git, you can always go back to any previous version. If you break something, paste this to Claude: "the dashboard broke, roll back to the last working version". Vercel keeps every deployment forever; you can instantly switch the live site to any prior version from Vercel's Deployments tab with one click.

When something breaks

The four things that go wrong most often, and what to do.

"My deployment failed in Vercel"

Click into the failed deployment in Vercel's Deployments tab. Scroll through the build log until you see red text. Copy the red text. Paste it to Claude with "Vercel build failed with this, fix it". Claude reads the error and patches the code. Then push again.

"My domain isn't loading / says 'not secure'"

Wait. DNS is slow. Check dnschecker.org with your domain. If it's not green everywhere yet, give it another hour. If after a few hours it's still broken: in Vercel, your domain page will show what's wrong (usually a wrong DNS record). Copy the exact error message into Claude and paste your DNS records (with secrets removed). Claude will tell you what to fix.

"The password prompt isn't appearing"

Two usual reasons: (1) you forgot to add the env vars in Vercel, or (2) you added them but Vercel hasn't redeployed since. Fix: in Vercel → Deployments tab → click the three-dot menu on the latest deployment → Redeploy. Or push any small change.

"I can't get past my own password prompt"

You typed the wrong username/password. Vercel → Settings → Environment Variables shows you what DASHBOARD_USER is set to (click the eyeball to reveal). For the password, you'll need to either remember it or replace it. To replace: edit the DASHBOARD_PASS value, save, then in Deployments tab → redeploy. New password active in 60 seconds.

When in doubt

Paste the symptom to Claude

You don't have to diagnose anything yourself. "The site shows X, what do I do" is enough. Claude can read the build logs, the config, and the error, it's a much better debugger than I am at 3am.

Quick Reference

Glossary, what the words mean

GitHub
Where your code lives. Like Google Drive but for software folders. Free. Private repos mean only you (and Vercel, once you authorize it) can see them.
Vercel
The host. The company whose computers actually serve your site to anyone who visits the URL. Free tier covers everything in this guide.
Deploy
"Make the changes live." Vercel takes your latest code from GitHub, builds it, and puts the result on the internet at your URL.
DNS
The internet's address book. Tells browsers where to send requests for your domain. Lives at your domain registrar (Cloudflare, Namecheap, etc).
Registrar
The company you bought your domain from. Manages DNS for that domain.
CNAME / A record
Types of DNS records. A CNAME points one domain at another domain. An A record points at a specific IP address. Vercel tells you which one to add.
SSL / HTTPS
The padlock in the address bar. Vercel sets this up automatically and for free. You don't have to think about it.
Env var
Environment variable. A secret value (like a password) that your code reads at runtime. Stored in Vercel, never in your code, never on GitHub.
Push
Upload your latest code to GitHub. The git command. Triggers Vercel to redeploy.
Commit
A snapshot of your code. One unit of change with a message. You commit before you push.
